## Context

[Issue #1](https://github.com/wangjc683/galley/issues/1) from `@lauchiwa` (Windows uv user) — Galley's health check looks for `.venv/bin/python` but `uv venv` on Windows creates `.venv/Scripts/python.exe`. The report is accurate, but it's the visible tip of a structural problem: alpha.1 / alpha.2 forced every user to wire up a Python interpreter themselves, with a static allowlist that can't grow fast enough to cover the universe of Python managers (system Python.org / Homebrew / pyenv / conda / asdf / uv / pdm / poetry × multiple OS layouts). Mac dogfood hit the same wall on first prod build — `launchd` PATH resolved `python3` to the macOS-bundled 3.9.6, no GA deps, bridge died on import. JC reports multiple group users also got stuck on this step or worked around it via friction.

The fix space had two candidates:

1. **Patch the allowlist** (alpha.3 stop-gap): add `Scripts/python.exe` rows, ship in days, doesn't help conda/pyenv users. Maintainable cost: a chronic illness — every Python manager that doesn't fit becomes a new issue.
2. **Bundle Python with Galley** (v0.1.1 architectural): include CPython + GA core deps in the .dmg / .exe. ~80MB binary, zero user-side Python config. Single fix.

GA's actual deps surprised the planning — initial planning assumed deps were heavy (anthropic SDK / openai SDK / lxml / etc.) and bundle would require runtime `pip install` (network + 30s wait). Measured reality: GA core deps total **0.93 MB of wheels** (`requests + beautifulsoup4 + bottle + simple-websocket-server` + their transitives), all near-pure-Python. The "wait + network" cost evaporated — we just stage them in `site-packages/` at build time.

The unsigned-release stance also got formalized this session — JC: "Galley 目前阶段不考虑 dmg 签名，都先以未签名形式发布，作为个人开源项目，问题不大." This is consistent with alpha.1 / alpha.2 actual behavior (release.yml never had codesign/notarize steps) but had never been an explicit project policy.

## Decisions

1. **Bundle CPython 3.11.15 (PBS `install_only_stripped`) + GA core deps** into every Galley release. Per-arch binary, never universal2 (JC: split per-arch is final).
2. **Version label = `v0.1.1-alpha.1`** despite the architectural-minor-bump-worthy diff. Alpha tag carries the "still polishing" signal; reserving v0.2 for the Windows release narrative was JC's call. Architectural intent gets surfaced in release notes + this devlog, not the version string.
3. **Skip alpha.3 stop-gap**. Win Scripts/python.exe patch would be throwaway — v0.1.1 retires the entire static allowlist. lauchiwa waits a few days, gets the real fix.
4. **Unsigned release as Pre-v1.0 project policy** — codified in CLAUDE.md. CI doesn't call `codesign` / `notarytool`; users right-click → Open to bypass Gatekeeper. Release notes should explain.
5. **Single-arch bundle output layout** — `python-bundle/python/` (no per-arch subdir). Tauri builds one arch per invocation; CI runs `bundle-python.sh <arch>` per matrix entry, overwriting. No JSON rewriting hack needed.
6. **Default `gaConfig.useExternalPython = false`** — bundled is the path. External is the escape hatch (Settings → Runtime advanced toggle, landing in Phase 3). Hydration migration: alpha.2 configs get `false` by default so upgrading users automatically transition.
7. **Dev mode unchanged** — `pnpm tauri dev` doesn't have `$RESOURCE/python/` at runtime (bundle.resources mappings are build-time only). bridge.ts forces external in dev (`!import.meta.env.PROD`), preserves current dev workflow.
8. **Mac Intel into CI via cross-compile on macos-15** (arm64 runner). Retires the alpha.2-era "JC local Intel build + manual upload" pattern. Phase 6 work.
9. **Two `python-bundled-*` aliases** — `python-bundled` (Mac/Linux, `bin/python3`) + `python-bundled-win` (`python.exe` at bundle root). PBS install_only layout differs by OS.
10. **`scripts/bundle-python.sh` ships as the canonical bundler** for local + CI. Output deterministic enough (pinned PBS release + pinned dep versions) that CI cache key = script hash + GA deps array.

## Rejected alternatives

- **Runtime `pip install` first-launch** — abandoned once GA deps measured at <1MB wheels. Network dependency + 30s wait for no real reason.
- **Universal2 (fat) Mac binary** — JC rejected, "我确定肯定是分 binary，这一点定了". Doubles binary size (~140MB single bundle vs ~85MB split each) for zero benefit on a project that already split-binaries per arch.
- **Tauri sidecar mechanism for Python** — sidecar is per-binary, but we ship a directory tree (`bin/`, `lib/python3.11/`, `share/`, etc.). Resources/ is the correct primitive; sidecar was a wrong path that surfaced briefly.
- **python.org Windows embedded zip** — smaller (~10MB) but stdlib incomplete and no pip. PBS Win `install_only_stripped` is the same source-of-truth as Mac, ABI consistency wins over 15MB.
- **Bundle with codesign + notarize this round** — explicitly out of scope. Apple Developer Account workflow + CI secrets setup + entitlement plist + notarytool credentials = independent engineering task, would double v0.1.1 scope. Pre-v1.0 unsigned policy formalized instead. Revisit at v1.0 or when user-pain crosses threshold.
- **`v0.2.0-alpha.1` version label** — semantically more accurate (architectural change earns a minor bump). JC preferred keeping v0.2 reserved for the Windows-release narrative, so v0.1.1-alpha takes the architecturally-loose alpha-tag latitude.
- **Per-arch bundle subdirs (`python-bundle/<arch>/python/`) + CI JSON rewriting** — POC iteration; replaced by single-output-location after recognizing Tauri only builds one arch at a time. Cleaner, no JSON munging.
- **Hydration migration default `useExternalPython = true`** — initial pass picked this (preserve alpha.2 behavior). Wrong: defeats the upgrade benefit. Flipped to `false` so v0.1.1 users get bundled automatically.

## Phase 0 POC verification

End-to-end smoke on JC's Intel Mac with unsigned `.dmg`:

```
$ pgrep -fl workbench_bridge
12459 /Applications/Galley.app/Contents/Resources/python/bin/python3 -m bridge.workbench_bridge --ga-path /Users/inkstone/Documents/GenericAgent --session-id s-mpak6zkj-vyee --llm-no 1
```

- ✅ Unsigned `.dmg` installs (right-click → Open absorbs Gatekeeper, internal Python binary inherits trust via .app quarantine cascade)
- ✅ Tauri `$RESOURCE` macro in `shell:allow-spawn` cmd resolves correctly to `<.app>/Contents/Resources/`
- ✅ `python-bundled` capability alias dispatches to bundled binary, not system Python
- ✅ Bundled Python `import agentmain` succeeds (GA core dep import chain works against staged site-packages)
- ✅ Bridge IPC + GA tool dispatch + UI rendering — full chain operational
- ✅ JC sent messages, GA responded normally

False-start (POC iteration cost): JC first reported all 5 bridge processes were spawning `/Library/Frameworks/Python.framework/Versions/3.14/...` instead of bundled. Root cause: he had built `tauri build --target x86_64-apple-darwin` into a new `.app` artifact but Applications/Galley.app was still the alpha.2 install (replacing-via-dmg silently didn't replace because Finder's dialog wasn't surfaced). Cleanup steps: `pkill -f workbench_bridge`, `rm -rf /Applications/Galley.app`, `cp -R target/.../release/bundle/macos/Galley.app /Applications/`, verify `Resources/` contains `python/` directory. After clean install, `pgrep` showed bundled path.
