## Decisions

### 1. Multi-session polish 4 项收尾

**per-session LLM**：之前 `llms` / `llmDisplayName` 在 store 顶层全局。N-active 下各 bridge 跑各的 LLM，全局字段会被最后一个 `ready` event 覆盖、切换 session 时显示错乱。搬进 `SessionRuntime`，顶层字段降为 active session 的 projection。`replaceLLMs(sessionId, llms)` 改 per-session 签名。

**title 派生**：`appendUserTurn` 首次提交时检查 `session.title === "新对话"`，截前 30 字作 title（带 `…` 截断标记），best-effort 持久化。已被用户改名的 session 不会被覆盖。

**summary 写入**：`bumpSessionAfterTurn(sessionId, summary)` 把 GA 的 turn summary 组成 `第 N 步 · {one-liner}` 写到 `session.summary` + SQLite。

**set_llm 接线**：CommandPalette `onSwitchLLM(idx)` 直发 `set_llm` IPC，bridge 回程 `llm_changed` 自动刷新 per-session state。

### 2. Session Restore: turn_index = turnCount + 1 + ready 事件触发 replay

GA 一对一 pairing（每条 user message 触发一个 turn 周期），所以 store 端可以预测 user message 的 turn_index：`(session.turnCount ?? 0) + 1`。`appendUserTurn` 时 fire-and-forget 写一条 user row 到 messages 表（sequence=0），turn_end 写 assistant row（sequence=1）—— sequence 差异避免同 turn 内排序不稳定。

历史恢复链路：
- **UI 复原**：`activateSession(id)` 看 runtime fresh + session.turnCount > 0 → `restoreSessionTurns(id)` 读 SQLite messages → `rowsToTurns` 转 Turn[] → 写进 runtime
- **GA backend 复原**：IPC `ready` event 检查 session.turnCount > 0 → `replayHistoryToBridge` 读 SQLite → 转 ConversationMessage[] → 发 `load_history` IPC。Bridge 命令队列 FIFO 保证 load_history 在用户 `user_message` 前处理

V0.1 简化：bridge 的 `_load_history` 把 content 包成单个 text block，丢失 GA 结构化 tool_use block。能继续 prose-y 对话，完整 fidelity 是 PRD §10 的 open item，等 per-backend adapter。

### 3. LRU 5 alive bridges

`_lruOrder: string[]` module-level，最近活跃在末尾。`spawnBridge` 成功后 + `activateSession` 命中已 alive bridge 时 → `_lruTouch`。每次 spawn 后 `_enforceLRUCap`（fire-and-forget）shutdown 最老的 non-active bridge 直到 `length <= 5`。

active session **永远保护**——suspend 用户在看的 session 是最糟 UX。`shutdownBridge` + `onClose` 都 `_lruRemove` 避免泄漏 phantom 条目。

Suspended session 视觉上 sidebar 行还在；用户切回 → `activateSession` 检测 `bridgeStatus === "closed"` → `spawnBridge` → ready → load_history（Task 3 链路）自动恢复 context。

LRU_CAP = 5 不是 PRD 规定，是「today 的 hot working set」量级猜测；可调。

### 4. Settings path picker：只 GA Path，Python 字段只读

`gaConfig: { python, gaPath, bridgeCwd }` 存 prefs key `ga_config`。`setGAConfig(partial)` merge + setPref + 同步 runtimeInfo + push toast 「已保存路径配置 · 重启 Workbench 才能让现有 session 生效」。`activateSession` spawn 时用 `get().gaConfig` 取代 hardcoded DEMO 路径。

**Bridge Python picker 故意不接**：Tauri `shell:allow-spawn` capability 只允许 `python3` / `python` 两个 alias（PATH 解析）；用户自选绝对路径会被 spawn 时拒绝。picker 看似工作实则失效。改 SettingsRuntime 让 Python 字段 `readOnly` + hint「使用 PATH 中的 python3 · 自定义路径需编辑 capabilities/default.json」。诚实 > 假 picker。

### 5. Onboarding 真 validation：fs.exists 5 项

`lib/onboarding-validation.ts`：
- `validateGAPath(rawPath)`: `~` 展开 + `fs.exists(path)` + `fs.exists(path/agentmain.py)`，三种 terminal outcome (`not-found` / `missing-agentmain` / `ok`)
- `runHealthChecks(path, onUpdate, signal)`: 5 项 sequential fs probe（GA path / agentmain.py / mykey.py / memory/ / assets/），220ms 间隔走「pending → running → success/warning/failed」，AbortController 支持中途取消
- mykey.py + memory/ + assets/ 用 `warnOnMissing` 处理「用户还没配置」case，warning 而非 failed

不跑 Python smoke test（spawn 一个 python 进程 + 额外 capability + 时间成本，对 fs check 已能覆盖的判断没增加价值）。Import 错误等 first bridge spawn 通过 `error` IPC 事件浮出。

App.tsx `onComplete(gaPath)` 调 `setGAConfig({ gaPath })` 把 onboarding 结果持久化（之前只 setScreen("empty") 丢路径）。

### 6. macOS bundle：bridge/ 作 Tauri resource

`tauri.conf.json` `bundle.resources: { "../../bridge": "bridge" }` 把 `bridge/` Python 包复制到 `.app/Contents/Resources/bridge/`。`spawnBridge` 用 `import.meta.env.PROD` 区分：
- prod: cwd = `resourceDir()`（指向 Resources/），`python -m bridge.workbench_bridge` 找到包
- dev: cwd = args.bridgeCwd（workbench 项目根，bridge/ 在 desktop/ 旁边）

代码签名 + notarization 不在 V0.1 scope，用户右键 → 打开走一次 Gatekeeper 即可。

### 7. Dogfood UX 多轮（7 轮反馈）

第一次真跑 dev 后 user 报的问题，按轮次：

**Round 1**：
- Composer 1 行 textarea，多行输入看不见 → auto-grow（rows=2 起步 + useEffect 监听 scrollHeight 撑高，280px ≈ 10 行封顶后内部滚动）
- LLM 切换点 pill 弹整个 CommandPalette → Radix Popover 内联下拉，点击直发 `set_llm`，ChatGPT 同款 UX
- Sidebar 无 Archive 路径 → 装 `@radix-ui/react-context-menu`，右键 menu → archive

**Round 2**：
- LLM popover caret ▼ 但菜单往上弹 → 改 `CaretUp` ▲（窗口底部没下方空间，永远往上展开是对的）
- Archive 操作无反馈 → push info toast 「已 Archive · {title}」
- 「新对话」累积 → 双管齐下：(a) 移除 App.tsx auto-create useEffect，改 EmptyState.onSubmit lazy create；(b) hydrateFromDB 加 `deleteEmptyNewSessions()`（title='新对话' AND turn_count=0）+ `deleteDemoSessions()` 清 v0.1 DEMO 占位

**Round 3**：
- Archive toast 显示「操作未能完成」→ `defaultTitle(error)` 给 category=business 永远 fallback 这个文案。AppError 加 optional `title` 字段，archive 显式传 `title: "已 Archive"`。setGAConfig + createSession 同类问题一起修
- Sidebar New Chat 点一次建一个 session → 改 lazy：`setActiveSession(undefined) + setScreen("empty")`，submitOnEmpty 真按 send 时才 createSession

**Round 4**（thinking placeholder 文案/气质）：
- 「LLM Running (Turn 1) ...」+ 大段 mono uppercase TURN N 占两行，理工监控感太重 → TurnMarker `font-mono uppercase tracking-[0.12em]` → `font-serif italic text-ink-muted`；MainView placeholder 合并 TurnMarker + 思考中… 成单行（DESIGN §4.3 原意就是单行）

**Round 5**（真正的 root cause）：
- 「LLM Running (Turn 1) ...」是 GA `agent_loop.py:49` 每个 turn print 到 display queue 的内容，bridge 转发 → desktop 当 LLM 输出渲染。所有 GA 官方 frontend（dcapp/tgapp/qtapp/stapp 等）都有 regex strip 它，desktop 漏了 → `LLM_RUNNING_MARKER` regex 加进 `cleanFinalAnswer` + `cleanPartialContent`

**Round 6**：
- 「第 N 轮」中文里「轮」=「对话回合」，user 发第 2 个 message 看到「第 1 轮」会困惑 → 全文重命名「第 N 步」（步骤/推理子步骤，避歧义）。三处统改：TurnMarker / thinking placeholder / sidebar `session.summary`

**Round 7**（sidebar 三状态）：
- 提议把 sidebar 状态拆双维度：运行状态 ⊥ 已读状态。spinner（运行中）/ static icon + brand dot + bold title（跑完未读）/ static icon（跑完已读）。`002_add_has_unread.sql` migration + `bumpSessionAfterTurn` 触发 + `setActiveSession` 解除 + 持久化到 SQLite

**Round 8（bug fix）**：
- Sidebar 看得到 summary，主对话框看不到 → 根因 `turnFromTurnEnd` 参数类型没声明 `summary` 字段，TS 静默丢了 event.summary。AgentTurn 加 summary 字段 + TurnMarker 支持可选 summary，渲染成 `第 N 步 · {summary}` 跟 sidebar 二行预览一致

## Rejected alternatives

### Multi-session polish

- **全局 llms（保留现状）**：N-active 切 session 时显示其它 session 的 LLM，明显错乱。per-session 是正确架构。

### Session Restore

- **bridge 端持久化 history**：让 bridge 自己读 SQLite 不通过 IPC。但 bridge 是 dumb subprocess，desktop 是 SQLite owner，单向数据流更清晰；IPC `load_history` 已有 schema，直接复用。
- **load_history 后等 `history_loaded` 再让 user 发消息**：bridge 命令队列 FIFO，load_history 必然先于 user_message 处理，不需要 desktop 端阻塞。

### LRU 5

- **1-active**：上一轮直接被 user 一票否决（GA 用户多 session 并行跑长任务）。
- **保留所有 alive**：20 个 bridge ≈ 3GB RSS，桌面用户机器扛不住。
- **active 也能 suspend**：user 在看的 session 突然「闪现 spawning…」是最糟体验。
- **数字 badge（"3 条新回复"）**：对 chat session 没意义——切过去就一次看完累积内容。boolean 够用。
- **error IPC 也触发 unread**：error 已 push 全局 toast 立刻到达用户，再标 unread 信息冗余。

### Settings path picker

- **Python 也 picker**：Tauri capability 限制让自选路径失效，做了等于骗用户。改只读 + hint 是诚实做法。
- **改完路径自动 shutdown 所有 bridge 重启**：会打断用户正在跑的 turn。DESIGN §9 明确「重启 Workbench 才生效」，toast 提示即可。

### Onboarding validation

- **跑 python smoke test（import agentmain）**：要 spawn python + 时间 + 额外 capability。fs.exists 已能覆盖 90% 验证；import 错误等 first bridge spawn 由 error IPC 浮出。
- **mykey.py 缺失 → failed**：mykey.py 是 user-supplied + .gitignored，新用户 attach GA 时还没配是常态。`warnOnMissing` 处理成 warning 不阻断 onboarding。

### macOS bundle

- **Python sidecar（预编译 binary）**：bridge 是 Python 包，sidecar 适合 native binary。Tauri resource 直接复制 bridge/ 目录最简洁。
- **excludes __pycache__ / tests/**：可以做但浪费几 KB 无害，先不优化。

### Dogfood UX

- **LLM popover 强制 side="bottom"**：Composer 在窗口底部，Radix 会自动翻转到 top；显式 `side="top"` 锁住更可控。caret 同步改 ▲ 解决视觉预期。
- **Sidebar 内部过滤 archived**：双重过滤无害但分散逻辑；改 App.tsx visibleSessions 单点过滤，Sidebar + CommandPalette 共用。
- **Archive 后自动 shutdown bridge**：用户可能 archive 视觉清理但希望在跑的 turn 跑完。让 LRU 自然回收 / app exit 清理，不主动 kill。
- **Auto-create empty session on screen mount**：第一版的做法，每次开 app 就累积一个空「新对话」。lazy create 是对的——user 真按 send 时才建。
- **「第 N 轮」继续用**：「轮」中文歧义太大，必须改。
- **去掉 step 编号只显示「思考中…」**：复杂任务跑 10 个 step 时用户看不见进度，损失信息。
- **TurnMarker summary 折叠（accordion）**：summary 通常 < 60 字短文本；跟 sidebar 默认显示不一致；多一步交互负担。直接显示是对的，user 还要折叠再加。

## Open questions

1. **messages 表加 summary 列**：当前 `final_answer` + `thinking` 列存在，summary 没列；restore 后 conversation TurnMarker 失去 summary（sidebar 还在，独立 `sessions.summary` 列）。Migration v3 + persistTurnEndToMessages 写 + rowsToTurns 读，三步事，看用户实测是否觉得这点 fidelity loss 影响大。
2. **`bridge.resources` 排除 `__pycache__/` + `tests/`**：复制了无害但浪费几 KB；Tauri v2 resource map 支持 glob 排除否？没查。
3. **代码签名 + notarization**：unsigned distribution Gatekeeper 一次性 prompt 可接受，正式发布前需要 Apple Developer 账号 + notarytool 流程。这是 release process 工作不是工程，等 V0.1 release 时再面对。
4. **`load_history` content 包成单 text block 丢 tool_use 结构**：PRD §10 open item，per-backend adapter 工作。GA backend 类型扩展时跟进。
5. **LRU_CAP 5 是否合适**：拍脑袋数字。实际用一段时间看是不是太紧（用户常感觉「咦怎么又 suspend 了」）或太松（资源压力）。
6. **Onboarding `expandHome` 路径手输 robustness**：picker 返回的总是绝对路径；只有手敲 `~/Documents/...` 才走 `homeDir() + slice(1)` 路径。少见但要处理 `~user/...` 这种 shell expansion 形式不到位时 user 会困惑。
7. **Bridge Python picker 长期方案**：要么改 capability 允许自定义 cmd（不安全），要么走 Tauri command（Rust 侧 spawn 绕过 capability）。V0.2 决定。
8. **bridge 启动失败 / GA 不存在 时 onboarding 完成后该怎样**：当前 onComplete 写 setGAConfig 就走 empty screen。如果路径其实启动 bridge 也会失败，user 进 empty screen 看不到任何提示。等真出问题时再处理 retry 流程。
