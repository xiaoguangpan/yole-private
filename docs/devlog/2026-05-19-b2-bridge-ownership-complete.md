## Context

B1 (2026-05-18) shipped the Rust authority layer's read skeleton + a working `yole` CLI binary. B2's job: own the runtime side — Python runner subprocess lifecycle, socket transport for CLI write commands, schema migrations for origin tracking, and a thin GUI shim that doesn't know it's been migrated. Constraint: **0 user-facing regression** through M1-M6; dogfood validation is the M7 final gate.

What got shipped in this session (chronological):

- **M1 RunnerManager scaffold** (`59176a2`) — `core/src/ipc.rs` mirrors `runner/ipc.py` (17 events + 12 commands, serde `tag = "kind"` + camelCase wire format). `core/src/runner_manager/{mod,process,manager,error}.rs` ports the bridge-owner prototype's `BridgeProcess` to production: typed `BroadcastItem::{Event(Box<IpcEvent>), Malformed(String)}` channel, per-session HashMap with LRU eviction (active + agent_running double protection), rolling 8-line stderr buffer, three typed error enums (`RunnerSpawnError` / `SendCommandError` / `ShutdownError`). 35 new tests (26 unit + 9 integration spawning a mock Python runner from a tempdir).
- **M2 bridge.ts thin shim** (`82ffedc`) — 5 `#[tauri::command]`s wrap `RunnerManager` (`spawn_runner` / `send_to_runner` / `shutdown_runner` / `kill_runner` / `runner_stderr_tail`; +1 `shutdown_all_runners` for app cleanup). RunnerManager held as Tauri app state via `Arc<>`. Per-session emit task subscribes to broadcast + fans events to GUI through `runner-event` / `runner-malformed` / `runner-closed` Tauri events. `gui/src/lib/bridge.ts` rewritten: function signatures byte-identical (B2-I1), body switched from `@tauri-apps/plugin-shell` to `invoke()` + `listen()`. `useAppStore.ts` 0 changes. CI gate (`check.yml`) gained `cargo test -p yole-core` + `-p yole-cli` per matrix target.
- **M3 socket listener** (`19e33d6`) — `core/src/socket_listener.rs` (~430 lines): Unix domain socket at `$TMPDIR/yole-$UID.sock` (0600 perm, hand-rolled `extern getuid()` to skip libc dep) on macOS/Linux; Windows named pipe at `\\.\pipe\yole-$USERNAME`. Function-level `#[cfg(unix)]` / `#[cfg(windows)]` instead of enum unification (Windows pipe semantics don't fit; stream-level generic handler reuses dispatch). NDJSON request/response framing + stable error discriminants (`invalid_args` / `not_found` / `db_unavailable` / `unknown_command` / `schema_mismatch` / `not_implemented` / `idle_timeout` / `internal`). Race detection at startup: try-connect with 200ms timeout → either bail (another instance) or unlink stale + bind fresh. `SocketGuard` Drop unlinks file at app teardown; dormant variant no-ops. 90s idle timeout. `tauri::async_runtime::block_on` in setup hook (bind is ~ms). 19 tests (12 unit + 7 integration round-tripping through a private tempdir socket).

CLI subcommands `yole session send <id> "..." - **Dogfood fixes** (`b087f22`) — JC's first dev-mode session caught 3 distinct bugs after M2 went live. (1) **`ReadyEvent.availableLLMs` silently dropped**: Python emits `availableLLMs` (LLM acronym caps preserved); Rust's `#[serde(rename_all = "camelCase")]` mapped `available_llms` → `availableLlms` (single capital), parse fell through to `#[serde(default)]` and re-serialized with the wrong wire name. Fix: explicit `#[serde(rename = "availableLLMs")]` + regression test `parse_ready_event_carries_available_llms`. (2) **Capability-alias `gaConfig.python` not translated**: v0.1 stored `gaConfig.python` as a Tauri shell-capability alias name (`python-brew-arm` etc.); M2's `Command::new(args.python)` does a plain OS exec lookup which silently fails. Fix: `bridge.ts:resolvePythonPath` now translates aliases via `findCandidateByAlias` (same table the v0.1 probe uses). (3) **spawn error didn't say which python failed**: `io::Error { kind: NotFound }` doesn't carry the program name. Fix: `RunnerProcess::spawn` maps `io::Error` → `RunnerSpawnError::PythonNotFound { detail: "no such file: '<path>' (set Settings → Python or check PATH)" }`.

End state: GUI tx/rx through Rust runner_manager (same UX, different transport); CLI write commands wired through socket transport end-to-end (CLI → socket → trait → DB + runner_manager); origin tracking persisted on every B2+ write; public contract (`agent-api.md`) and wire protocol (`ipc-protocol.md`) both documented. **Code complete; dogfood 1-week stability + perf-gate run deferred to next session.**

## Decisions

| # | Decision | Why this not that |
|---|---|---|
| 1 | **`bridge.ts` function signature locked through M2** (B2-I1) — body rewritten but the interface byte-identical | useAppStore.ts (2858 lines, 6 months of dogfood UX teaching) stays unchanged through B2. Slice refactor is B3's job; bundling it with the Rust ownership migration would compound risk. B3 then assumes bridge.ts is a thin shim it can safely thread through. |
| 2 | **5 Tauri commands, not 4** (playbook stub said 4) | The playbook listed spawn/send/shutdown/kill. M2 implementation added `runner_stderr_tail` (pull-style API for the abnormal-exit toast that pre-M2 useAppStore expected from each-stderr-line push) + `shutdown_all_runners` (called from JS cleanup hook on quit). Both are pure helpers; neither breaks the "5 commands" mental model agents need. |
| 3 | **stderr surfaces via pull-style `runner_stderr_tail`**, not per-line Tauri events | Per-line stderr events in multi-session high-load streaming would have meant N events/line × M sessions Tauri event traffic. The only consumer is the onClose abnormal-exit toast, which only needs the last 8 lines — pull on demand at close time hits the same UX target with no event overhead. |
| 4 | **Function-level `#[cfg(unix/windows)]` for socket platform abstraction**, not enum unification | Initial plan was `enum Listener { Unix(UnixListener), Windows(NamedPipeServer) }`. Implementing it surfaced that Windows pipe semantics (one server instance per client, must create a fresh `ServerOptions` after each `accept`) are fundamentally different from Unix `accept()`. Stream-level generic handler (`handle_stream<R, W>`) gets reuse where it matters; listener-level union doesn't pull its weight. |
| 5 | **Hand-rolled `extern "C" fn getuid()` in `socket_listener.rs`** — no `libc` or `nix` crate | Six characters of code; pulling in either crate would add ~5 transitive deps for one syscall. Same rationale as B1's hand-rolled `YoleError` Display (no thiserror) and the chrono-free ISO 8601 formatter in `SqliteYole::send_message`. |
| 6 | **`tauri::async_runtime::block_on` in setup hook** for socket bind | Bind + race detection is ms-scale; spawning an async task lets startup proceed without the socket being live, which means CLI clients connecting in the first frame would race. Blocking the setup hook (already a sync function) for ~10-200ms is clean. |
| 7 | **`session.send` returns `{message, dispatch}` not just MessageBrief** | Stub playbook said "fire-and-forget, return MessageBrief". Implementation surfaced that persist and runner-dispatch are independent — a session whose runner is LRU-evicted or crashed will persist but not dispatch. The `dispatch: "dispatched" | "persisted_only"` field tells the CLI client which case it hit; still fire-and-forget (doesn't wait for `turn_start`), still agent-friendly. |
| 8 | **`session.watch --from=<event-index>` deferred to B4** | Would need a per-RunnerProcess ring buffer; broadcast channel is fan-out not history. The supervisor-reconnect use case (the only thing `--from` solves) is naturally a B4 concern when supervisor SOPs ship; ship M4 without it, take a simpler API. |
| 9 | **Migration numbering 006 / 007** for B2 origin fields (not 010+ as `invariants.md §I3` table suggested) | The §I3 hard rule is "no gaps in numbering". B1 didn't introduce any migrations (B1-I1: no write path), so 006 was the natural next free slot. Same PR updated §I3 table to reflect actual commit order — the phase-mapping in the original table was a planning hint, not a contract. |
| 10 | **`OriginVia` widened `{Manual, Cli}` → `{Gui, Cli, Supervisor, System}`** (technically breaking — but additive in practice) | The SQL CHECK constraint accepts 4 values; the B1-era enum had 2. There were zero wire callers emitting `"manual"` (B1 enum only existed as a trait signature placeholder), so the rename was a free upgrade. New code uses `Origin::gui()` / `Origin::cli()` convenience constructors; the helper `as_sql()` keeps the SQL bind aligned. |
| 11 | **`SqliteYole::send_message` persists + returns; socket layer dispatches** | Trait method only writes the row. The socket dispatch wrapper (`dispatch_session_send`) is the one that calls `manager.send_command` after a successful persist. Reason: keeps trait pool-testable in-memory (no manager dependency); transport layer composes persist + dispatch, agent CLI sees one round-trip. |
| 12 | **Manual sync between `runner/ipc.py` and `core/src/ipc.rs`** — no codegen | Two source-of-truth files for the IPC wire format. Adding codegen (e.g. derive-from-python) would cost a build-time tool + sync infrastructure for ~30 fields' worth of one-time copy. The `availableLLMs` acronym bug (caught in JC dogfood, fixed in `b087f22`) is exactly the kind of slip codegen would prevent — but codegen also locks both sides to the codegen tool's lifecycle. Manual sync + regression tests on the wire names is the lighter answer for now. |
| 13 | **B3 playbook upgraded same session** (`3002f76`, before M7) | JC opted to upgrade the B3 stub while M6 socket / agent-api context was fresh, rather than waiting until dogfood ended. M7 ceremony then just retires the B2 cursor and writes this devlog. B3 is the riskiest phase ahead (slice refactor of a 2858-line file with 6 months of UX teaching); having its playbook detailed before context decays was the right call. |

## Rejected alternatives

- **Per-line stderr events**: Tauri event traffic scales linearly with output volume; the only consumer is the abnormal-exit toast. Pull on close.
- **Codegen for IPC type sync** (`runner/ipc.py` ↔ `core/src/ipc.rs`): would have prevented the `availableLLMs` acronym bug, but adds a build-time tool + sync infrastructure that locks both sides to the tool's lifecycle. Bug was caught in first dogfood and trapped by a regression test — manual sync stays as the lighter answer.
- **Enum-unified `Listener { Unix, Windows }`**: Windows named-pipe semantics (server-per-client, recreate on each accept) don't fit cleanly. Function-level cfg with a stream-level generic dispatch handler shares the work that's actually shareable.
- **`schemars::JsonSchema` derives on every IPC type**: deferred to whenever agent-api.md actually needs auto-generated schemas. Current hand-curated tables in the doc are more readable.
- **`thiserror` for `RunnerSpawnError` / `SendCommandError` / `ShutdownError`**: hand-rolled `Display` + `std::error::Error` per `YoleError` convention. ~30 lines total; tracks the wire-discriminant for `serde::Serialize`.
- **`session.watch --from=<idx>` backlog support in M4**: deferred. Needs a ring buffer per RunnerProcess (broadcast channel is fan-out not history); supervisor-reconnect use case lands properly with B4 supervisor SOPs.
- **Tag-on-dogfood-end vs tag-on-code-complete**: JC chose code-complete tag. Dogfood happens between sessions; if regressions surface they get fixed in B3 prep, not as B2 patch releases. The tag marks "this is the code state CLI agents should target", which is true even before dogfood.
- **Splitting M4 + M5 into separate commits**: they're cohesive — M4 needs M5's `Origin` enum to construct the `created_via='cli'` write, and M5's `send_message` trait method only exists to serve M4's socket dispatch. One commit, smaller PR surface.
- **Bundling the dogfood fixes (`b087f22`) into M7 ceremony commit**: kept separate because they're a clear failure mode (with regression tests) worth surfacing in `git log` distinctly. M7 commit is the no-code devlog/docs/tag commit.

## Open questions

- **Dogfood 1-week stability period**: deferred to next session. JC will daily-drive on the M1-M6 build; if regressions surface they go into B3 prep work. Specifically watch for:
  - bridge crash → abnormal-exit toast still shows last stderr lines (M2 pull-mode regression risk)
  - LRU eviction still silent (the user shouldn't notice the 6th-session-opens-7th-closes mechanic)
  - `/btw` side-question routing through the same runner stdin path (B2 didn't touch this; verify no accidental regression)
  - `yole session send <id> "..." --supervisor=jc --reason=test` real wire path (first time agents can write into Yole from outside the GUI)
  - `yole session watch <id>` Ctrl-C cleanup (CLI socket Drop should unlink cleanly)
- **Performance gate** (`invariants.md §I7`): P1 first-token RTT + P2 streaming throughput vs the prototype baseline. Prototype measured `p99 RTT 614µs` and `4684 events/sec sustained`; B2 should be within 50ms / 10% of those. Measurement requires running dogfood scenarios first.
- **Socket file 0600 TOCTOU window** (~1ms between `bind` and `set_permissions`): real exploit path is narrow but exists. Hardening (`umask(0o077)` before bind + restore after) deferred to B4 along with the other hardening work.
- **`session.watch` backlog (`--from`)**: tracked as a B4 deliverable. RunnerProcess needs ring buffer changes to support; ship when supervisor SOPs need the reconnect-resume pattern.
- **CLI happy-path integration test**: M4 only ships the "Yole Core not running → exit 4" path test. Full round-trip (start real RunnerManager + mock Python runner + connect socket + send + watch + assert events) is meaningful but heavy; ship after dogfood validates the design first.
