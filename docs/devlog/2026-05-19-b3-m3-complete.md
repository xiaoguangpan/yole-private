## Decisions

### Perf baseline 框架（commit `7c648fa`）

- 5 个 metric 完整 spec：P1 first-token RTT / P2 streaming throughput / P3 CLI read RTT / P4 bridge spawn cold / P5 RSS plateau
- 自动测的 metric（P3 / P4）由 AI 跑；JC 配合驱动真 LLM 的 metric（P1 / P2）走 `scripts/perf-yole.py` 协调 send + watch。
- P5 (RSS plateau) 推到 dogfood 期间被动收集（不阻塞 M3 启动）。
- **gate 规则 codified**：P1 ≤ 前阶段 + 50ms / P2 ≥ 前阶段 × 0.9 / P3-P5 sanity check 无硬阈值。

实测数据（B2 baseline）：
- P1 first-token RTT: **3.15s short prompt / 5.36s long prompt**（5.36s 长 prompt 比 short 多 2s 是 LLM 推理负载差异，非 IPC 开销）
- P2 streaming throughput: **1.42 events/sec**（GA bridge chunks LLM 流为 ~700ms 一个 chunk，不是 token-by-token；14 个 deltas / 9.86s for 500-字 essay）
- P3 CLI RTT: 13-15ms mean (4 commands × 20 samples)；几乎全是 process startup + sqlx connection 开销
- P4 bridge spawn cold: **cited prototype 430ms**（RunnerManager 是 BridgeProcess productionized，subprocess 路径 B2 未改 → prototype baseline = B2 baseline）

### 测量过程暴露 3 个 measurement-time finding

1. **Script bug**: `readline()` 在 inner "wait 1.5s for trailing turn_progress" loop 里 block 死，timeout 检查在 outer 进不去 → 第一次 P1 跑卡 4 分钟。修复：换 `select.select()` non-blocking read。
2. **GA wire format**: `{"stream":"event","data":{"kind":"turn_progress","delta":"..."}}` — parser 要查 `data.kind` 不是 top-level，否则全部事件被忽略。
3. **GA emit ordering surprise**: 长 prompt 时 `turn_progress` (真内容) **早于** `turn_start` —— `turn_start` 是 metadata-commit 事件，长 prompt 时 fire at END of streaming。**P1 marker 必须用 `first_real_delta`** 不是 `turn_start`，否则长 prompt 会被 over-report 至 15s 而非真实 5s。

### 4 个 B2 latent bug fix（per commit 独立 commit）

#### Fix 1: CLI-origin user message GUI 不显（commit `4e7a6e6`）

**Symptom**：`yole session send --supervisor=...` 持久化消息 + dispatch bridge + agent 响应了（GUI 看到 "二"），但**用户提问本身**从未出现在 conversation view。

**Root cause**：GUI 的 `appendUserTurn` 只有 Composer send path 调用。CLI-origin message 持久化到 DB + dispatch bridge stdin 全 work，但**没有 Tauri event 告诉 GUI 把 row mirror 进 in-memory `turns` 数组**（rendering source of truth）。

**Fix**：Rust `socket_listener::dispatch_session_send` persist + dispatch 后 emit 新 Tauri event `user-message-persisted`；GUI App.tsx 加全局 listener；新 store action `appendUserTurnExternal` mirror `appendUserTurn` 减 `persistUserMessage` 调用（DB 已经写了）。

#### Fix 2: 每 session LLM 选择不持久化（commit `9c36f42`）

**Symptom**：用户在某 session 切换 LLM、关 Yole、重开点开该 session，bridge 重新 spawn 用 mykey.py `is_default=True` 默认 → 用户的选择丢失。

**Root cause**：`sessions.llm_index` / `llm_display_name` 列在 migration 001 早有，**但 `persistSession` 硬编码 `null, null` 带注释 `wired in #10` 没人接上**。一个 ship-on-day-1 的 latent bug。

**Fix**：纯接线 — Rust `SessionBrief` 加两字段 / GUI Session type 加两字段 / loadSessionsViaCore + persistSession 改 binding / `replaceLLMs` 调用时 mirror 到 session row + persist / `activateSession` 在 respawn existing session 时 pass `llmIndex` 给 spawn。

#### Fix 3: LLM picker flash root cause（commit `d6a096f`，原 `2779baa` 修法不够）

**Symptom**：点开旧 session 时 LLM picker 先快速闪过 DEMO_LLMS 的 "Claude Sonnet 4.5"（即使该用户从未配置过 Sonnet 4.5），约 ~430ms 后才纠正到真模型。

**第一次修法（`2779baa`）不够**：只在 setActiveSession 返回值的 spread 后覆盖顶层 state.llms，但 `_runtimes[id]` 自己还是 emptyRuntime() with DEMO_LLMS。**下一个 applyRuntimeUpdate（setBridgeStatus inside spawnBridge）** 读 `_runtimes[id]`，projectionFrom 又把 state.llms override 回 DEMO_LLMS。flash 复发。

**第二次修法（`d6a096f`，真 fix）**：从源头给新 runtime 的 `llms` 字段 pre-seed 真值（用 hydrate-cached LLM list + persisted session display name），_runtimes[id] 从 t=0 就 authoritative，subsequent applyRuntimeUpdate 不会回退。

**附带 fix「老 session 永远卡 Sonnet 4.5」**：同根因 — bridge 若不 emit ready，`replaceLLMs` 永远不被调用，`_runtimes[id].llms` 永远 DEMO_LLMS。Pre-seed 让 picker 在 bridge 未 ready 时也显示真模型。

#### Fix 4: Picker click 在 spawning 期间 silent drop（commit `317e816`）

**Symptom**：上一个 fix 让 picker 从 t=0 显示真模型，用户自然在 bridge 还在 spawning 时就点切换 → popover 关闭但 pill 名不变 = 「点了没反应」。

**Root cause**：App.tsx 的 `onSelectLLM` handler `if (bridgeStatus === "connected") { void sendIPCCommand(...) }`。Spawning 期间状态是 `"spawning"`，gate 卡住，click silent drop。

**Fix**：放宽 gate 到 `connected || spawning`。`sendIPCCommand` 内部已经有 `_bridgeClients.get` 保护（无 client 时只 warn 不 crash），gate 严格只是 defensive overkill 造成 UX 反直觉。

## Rejected alternatives

- **M3a 同时迁 spawnBridge / shutdownBridge 等 bridge 行为**：被 sub-plan §2 第一性原理拒绝 — bridge lifecycle 跟 LLM concerns lifecycle 不同，混在一起 risk 显著，dogfood-able 颗粒度变粗。坚持 LLM-only first。
- **Cross-store onClose 走 Rust event 驱动**：M3 不动 Rust（B3-I4）。Rust 端不加 `runtime-updated` event，复用现有 `runner-event` / `runner-closed`。M5 messagesStore 落地时再做 event-driven refactor。
- **保留 useAppStore.replaceLLMs 作 deprecated shim**：违反 [B3-I3「no double-track」](../refactor/B3-store-slice.md#phase-invariants--b3-特有的硬规则)，single-commit 内 swap 所有 callers 才干净，shim 容易腐烂。
- **flash fix 只修 setActiveSession 顶层 projection**（commit `2779baa` 原方案）：fix 不彻底，applyRuntimeUpdate 再次 projection 时回退到 DEMO_LLMS。**真根因是 _runtimes[id] 本身 DEMO**，pre-seed 解决（commit `d6a096f`）。
- **LLM picker click gate 完全去掉**：保留 `connected || spawning` 而非 unconditional send — `sendIPCCommand` 内部有 `_bridgeClients` 保护，但显式 gate 提升 dev-time 可读性（"click 何时无效" 一目了然）。
- **CLI-origin user message 走 messages-appended 新 event**：[B3 emit catalogue](../refactor/b3-rust-emit-catalogue.md#2--messages-appended) 计划的 `messages-appended` event 是 M5 范围。M3 阶段加单 purpose `user-message-persisted` event = 最小 surface fix B2 latent bug，M5 时可合并到 `messages-appended`。
- **N4 「M3 启动门 override 推广」**：N5 已明 reject — M3 改动 authoritative state + 订阅化（比 M2 风险高一个量级），gate 仍 enforce。本 session 实际走得是 **JC dev dogfood 已 cover 主要风险面** 的事件驱动 gate 解锁，不是 override。

## Lessons learned

1. **「Thin wrapper」描述是 playbook 调研时的盲点**：T3.5 写得很轻 ("spawnBridge / shutdownBridge / sendIPCCommand 都是 thin wrapper")，实际查代码 spawnBridge 是 140 行 + 复杂 onClose 跨 store 写。**M4 sub-plan 写之前先重读所有要迁的代码**，不能信 playbook 表面描述。
2. **flash 类 UI bug 有两层 root cause**：先修「显示层」（setActiveSession 顶层 projection 覆写）—— 不够；再追到「数据层」（_runtimes[id] 本身 DEMO）—— 真根因。**遇到 "fix shipped + 复发" 时立即追 underlying state**，不要假设第一次 fix 完整。
3. **B2 latent bug 在 M3 实施前一次性扫清是很值得的投资**：4 个 fix 累计 ~3h，但保证 M3 dogfood signal 干净。如果带 latent bug 进 M3，验证 regression 时会 "M3 改动 + B2 bug" 混在一起难定位（N4/N5 教训再次确认）。
