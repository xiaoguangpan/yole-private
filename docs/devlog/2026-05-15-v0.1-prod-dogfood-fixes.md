# 2026-05-15 — v0.1 prod-build dogfood fixes (Onboarding routing · Python probe · bridge stderr · Settings revisit)

**Date / Status**: 2026-05-15 / shipped (code-level)
**Related**: [vision pivot devlog](./2026-05-15-vision-pivot-to-orchestrator.md) (parallel docs work) · [GA baseline upgrade devlog](./2026-05-15-ga-baseline-upgrade-6bb3104-to-fc6b5ad.md) (shipped together) · CLAUDE.md → 项目宪法 (the four interface surfaces protected by Python probe constraint)

## Context

First end-to-end packaged-build dogfood after weeks of `pnpm tauri dev` development. Built `Yole_0.1.0_x64.dmg`, opened it, hit **four critical bugs in the first 30 seconds**:

1. **No Onboarding** — fresh DB, but app drops straight into the new-chat screen
2. **LLM picker shows demo fixture** (`GLM 5.1 / Claude Sonnet 4.5 / GPT 4o / Gemini 2.5 Pro`), not JC's real `mykey.py` config
3. **Send message → no response** (silent — no error, no toast, just frozen)
4. **Settings → "Re-run health check" button** does nothing

Each turned out to have an independent root cause. The unifying theme: **dev mode worked by accident**, riding on the terminal's PATH and dev panel's dev-only screen toggle. Prod removed both crutches simultaneously.

This devlog captures the root-cause walk + the fix set. Bundled in the same session as the [vision pivot](./2026-05-15-vision-pivot-to-orchestrator.md) and the [GA baseline upgrade](./2026-05-15-ga-baseline-upgrade-6bb3104-to-fc6b5ad.md), since all three landed on the same dmg test cycle.

## Decisions

### 1. Root causes per bug

| # | Symptom | Root cause |
|---|---|---|
| 1 | No Onboarding | `screen: "empty"` is Zustand default; **no code anywhere calls `setScreen("onboarding")`** on first launch. Onboarding components existed but were only reachable via the dev panel — production never had a path to them. |
| 2 | LLM picker is demo fixture | Fresh DB has no `llm_list` cache pref → Composer falls back to hardcoded `DEMO_LLMS`. `warmupLLMList()` would refresh it, but it dies because of #3. |
| 3 | Send message → silence | `DEMO_GA_CONFIG.python = "python3"`. Packaged `.app` launched from Finder gets launchd's minimal PATH (`/usr/bin:/bin:...`) — so `python3` resolves to `/Library/Developer/CommandLineTools/usr/bin/python3` (macOS-bundled 3.9.6). System python has none of GA's deps installed. Bridge spawns, imports `agentmain`, immediately `ImportError`s, dies silently. Dev mode worked because `pnpm tauri dev` inherits terminal PATH (with Python.org 3.14 / Homebrew). |
| 4 | Re-run health check no-op | Handler was a `console.info()` placeholder both at the Settings entry and the Command Palette entry. Never wired. |

### 2. Python probe system (the centerpiece fix)

**Constraint discovered while designing**: Tauri's `shell:allow-spawn` capability does **not** support regex `validator` on the `cmd` field — only on individual arg values. The `cmd` is a literal string (`$HOME` / `$DOCUMENT` etc. base-dir vars supported, but no globs).

This made the original idea of "let user pick any Python absolute path" impossible without a Rust-side custom command. For V0.1, we pre-register ~10 candidate paths in the capability allowlist and probe each at runtime:

```
$HOME/Documents/GenericAgent/.venv/bin/python      (GA project venv — most common)
$HOME/Documents/GenericAgent/venv/bin/python       (alt name)
/opt/homebrew/bin/python3                          (Apple Silicon Homebrew)
/usr/local/bin/python3                             (Intel Homebrew — what JC's Mac wins on)
/Library/Frameworks/Python.framework/Versions/{3.14,3.13,3.12,3.11}/bin/python3   (Python.org)
python3 / python                                   (PATH fallback)
```

For each, spawn `<python> -c "<validation script>"` with a 4s timeout. First "ok" wins; persist its alias to `prefs.ga_config.python`.

**Validation surface — mid-execution correction (critical to write down)**:
The probe's first iteration ran `import anthropic, openai; print('ok')`. I'd assumed those SDKs were GA's LLM client layer. **Wrong.** Probing JC's machine showed zero Pythons could pass — but bridge *clearly* worked in dev mode. Looking at `llmcore.py:1`:

```python
import os, json, re, time, requests, sys, threading, urllib3, base64, importlib, uuid
```

**GA uses raw `requests` to call LLM APIs directly. It does NOT import the `anthropic` or `openai` SDKs.** The actual validation surface is whatever `agentmain` imports — most concretely `requests`, `bs4`, `bottle` from `pyproject.toml`.

Corrected probe runs `sys.path.insert(0, gaPath); import agentmain; print('ok')` — the exact chain bridge will take. When `gaPath` is null (early-bird probe before Onboarding's path picker), falls back to `import requests, bs4, bottle`.

**Lesson worth keeping**: when designing a "validate the dep chain" check, look at what the actual target imports, not what you think it should import.

### 3. Onboarding routing on first launch

`hydrateFromDB` now checks for a saved `ga_config` pref. If absent → `setScreen("onboarding")` + skip the warmup LLM bridge (which would just fail in prod without a probed Python). Once a user completes Onboarding, the pref is saved and they never revisit this branch unless they reset their data dir.

`Onboarding.onComplete` got a second parameter `pythonAlias: string | null` — the probe's winning alias from StepHealth. App.tsx persists both `gaPath` and `python` via `setGAConfig`.

### 4. Bridge stderr surfacing

Failure mode #3 (silent death) was the worst UX trap of the four. Even if Python probing covers most users, *something else* will eventually fail — broken `mykey.py`, wrong path, GA deps half-installed. The user has to see what went wrong.

Added per-session rolling stderr buffer (`_stderrTails: Map<sessionId, string[]>`, last 8 lines). On abnormal bridge close (`code !== 0 && code !== null`), push a `category: "bridge"` toast with the last 3 stderr lines. Same for spawn-time `onError` ("binary not found" / "scope rejected"). Graceful shutdown (code 0 / signal) stays quiet.

This means an ImportError now surfaces as "Bridge 进程崩溃 · `ModuleNotFoundError: No module named 'requests'`" instead of the chat just freezing. Independent of whether the python probe succeeded — it's a generic safety net for anything that breaks downstream.

### 5. Onboarding window drag region

Two-layer pattern:
- **Outer root**: `data-tauri-drag-region` — empty padding around content is draggable
- **Inner 700px content wrapper**: `data-tauri-drag-region="false"` — opts back out, so text stays selectable, buttons receive clicks

Tauri's drag region attribute inherits down the DOM tree by default. Setting "false" on the inner wrapper is the standard escape hatch.

### 6. Settings → Health Check redesigned (Option A from the discussion)

**Discovery**: After wiring the Python probe into Onboarding StepHealth, JC noticed Settings's inline Health Check table now diverged from Onboarding's cascade — different rows, no animation, button label "Re-run health check" was hooked but did something different (just a python-only re-probe via `rerunPythonProbe`).

The static table in Settings was also **lying**: `DEMO_RUNTIME_INFO.healthChecks` is a hardcoded "5 success rows" fixture that no real probe ever updated.

Three options discussed:
- **A.** Settings button reroutes through Onboarding's StepHealth in a new "revisit" mode (skips Welcome / Attach, jumps straight to Health)
- **B.** Replicate the cascade inline in Settings (double-maintenance, the divergence JC just observed will recur)
- **C.** Keep current state (broken)

JC picked A. Implementation:
- `Onboarding` accepts `mode?: "fresh" | "revisit"` + `onCancel?: () => void` + `initialPath?: string`
- In revisit mode: starts at `health` step, Back becomes "取消" (calls `onCancel`), Continue becomes "返回 Settings"
- App.tsx snapshots `screen` before opening Onboarding takeover and restores it + reopens Settings on completion or cancel
- `StepHealth` gained `backLabel` + `continueLabel` props for the label swap
- Settings's Health Check section shrunk from "static table + small Re-run pill" to "one prose line + one button"
- Deleted `rerunPythonProbe` store action (revisit flow supersedes it)
- Deleted `RuntimeInfo.healthChecks` field + its demo seed (was the lying fixture)

The wider observation worth keeping: **"if you have to maintain two parallel UIs for the same operation, you're going to have a bad time"** — the divergence appeared within a single session of independent code edits. Single canonical UX, accept the full-screen takeover as the cost for "rare action, never lies".

## Rejected alternatives

- **`import anthropic, openai` as probe surface** — initial design; corrected mid-execution after empirical disproof. Documented above.
- **Rust-side `tauri::command` for spawn** — would lift the "pre-register paths" constraint and accept arbitrary user paths (conda env, pyenv, asdf-managed Python). Bigger refactor (~200 lines Rust, rewrite of `bridge.ts` event plumbing). Deferred to V0.2 / B refactor. The pre-registered allowlist + auto-probe pattern serves the 90% case (GA users on Homebrew / Python.org / project venv).
- **Bundle Python sidecar with `.app`** — 50-100MB bundle bloat. Solves the problem completely but the wrong trade for V0.1.
- **Document "your Python must be one of these paths" as user requirement** — terrible UX; user shouldn't have to think about where their Python lives. The probe + tutorial-on-failure carries the burden.
- **Settings inline Health Check cascade (Option B)** — discussed above; double-maintenance trap, tutorial modal stacking on Settings dialog gets weird.
- **Surface every stderr line as a toast** — would spam on noisy Python warnings (`SyntaxWarning`, `DeprecationWarning`, GA's own `[Debug]` prints). Constraint to "abnormal exit + last 3 lines" filters to actually-actionable failures.
- **Re-run health check in Settings auto-spawns Onboarding from Welcome** — heavy; user has to click through Welcome + Attach + Health for what's a single check. Revisit mode jumps directly to Health.
- **Validator regex on `shell:allow-spawn` cmd field** — checked the Tauri v2 schema; validator only applies to arg values, not the `cmd` itself. Forces the pre-register-paths approach.

## Open questions

- **Conda / pyenv / asdf / uv users**: their Pythons aren't in the pre-registered allowlist. Auto-probe fails for them, the `python-missing-anthropic` tutorial points them at creating a `.venv` in the GA dir — which is the recommended path anyway. But "I have a working Python in conda and want to point Yole at it" isn't supported in V0.1. **V0.2 + Rust-side spawn fixes this.**
- **Probe TTL / re-validation**: probe runs once during Onboarding and saves the alias. If the user later breaks their venv (e.g., `pip uninstall requests`), bridge spawn still uses the saved alias and fails. The Settings → "跑一次 Health Check" button is the recovery path, but it's user-initiated. Whether to auto-re-probe on spawn failure (or after N consecutive failures) is a V0.2 polish question.
- **Onboarding revisit + GA path change**: revisit mode pre-fills `path` with the saved `gaConfig.gaPath` but doesn't expose Step Attach (the path picker). If the user realized their GA path is wrong, they'd use Settings → Runtime → GA Path picker. Edge case, but worth flagging.
- **`python-missing-anthropic` tutorial id**: the name is now misleading (after the anthropic→agentmain correction). Renaming the id would touch the index in `onboarding-tutorials.ts` plus the `isTutorialId` type guard. Cosmetic only; deferred.
