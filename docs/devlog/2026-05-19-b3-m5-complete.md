## Context

B3 (useAppStore 拆 slice) 最后一个 per-session authoritative state slice。M3a/M3b 把 LLM + bridge lifecycle 迁 runtimeStore；M4b 把 session/project list 迁 sessionsStore；M5 把 **conversation state**（turns / pendingApprovals / agentRunning / inFlightContent / approvalDecisions / pendingAskUser / turnIndexOffset / currentTurnIndex + global userSubmitTick）迁 messagesStore。完成后 useAppStore 只剩 prefs 字段 + hydrate orchestrator，M6 prefsStore 可以把这一壳拆完。

JC explicit「在这个 session 直接跑 M5 implementation」破了 sub-plan 写的「M4b dogfood 1 天 + fresh session」启动门 —— sub-plan 本来推荐 paperwork + implementation 分两 session。本 session 跑下来证实合理，单 commit 1052/1160 LOC 实施过程没有需要急刹的 surprise。

## Decisions

1. **Single commit M5 落地**（sub-plan §2.2 预案对齐）—— 拆 M5a/M5b 4 个候选 seam 全部失败（B-1 切分意义弱 / B-2 ipc-handlers 既读又写不能拆 / B-3 违反 B3-I3 / B-4 activateSession 半成品）。实施实测：useAppStore.ts -966 LOC 整段 deletion + 23 call site swap，单 commit 改 9 文件，TS strict + cargo test 一次过。
2. **`deriveSessionStatus` 签名换成结构化 `MessagesView`**（不是 `PerSessionMessages`）—— `lib/sessions.ts` 不引用任何 store 模块，需要的字段只有 `pendingApprovals: { approvalId: string }[]` + `agentRunning: boolean` 两项。换结构化 view 避免 `lib/sessions.ts → messagesStore → lib/sessions.ts` 反向依赖。
3. **`activateSession` orchestrator 搬 sessionsStore**（sub-plan §1.4 Option B）—— 跨 sessionsStore + runtimeStore + messagesStore 三方，归属 active id owner (sessionsStore) 更合理。useAppStore 越接近空文件越好，M6 才能彻底删它。
4. **Module-level EMPTY singletons + Object.freeze**（sub-plan §3 T5.1）—— `EMPTY_TURNS` / `EMPTY_APPROVALS` / `EMPTY_DECISIONS` / `EMPTY_MESSAGES` 都是 `Object.freeze([] as T[]) as T[]` 双 cast pattern。`readonly never[]` 不跟 `Turn[]` 兼容，所以 cast hop 必要；Object.freeze 是真 runtime safety net。React 19 strict-mode getSnapshot 稳定 reference 关键。
5. **跨 slice 改用 static import**（不再 dynamic import）—— M4b 的 `clearSessionRuntime` 用 `await import("@/stores/useAppStore")` 是为了避开 M4b 期间双向 module-eval cycle。M5 之后 runtime.ts 早就 static import sessionsStore，pattern 已经允许 cycle —— sessions.ts static import messages + runtime 也跟 pattern 一致，更可读。Vite 的 ES module cycle 处理只要不在 module-top-level 触读，cycle 是合规的。
6. **Sub-file split per G11**（sub-plan §3 T5.1 子文件预案）—— 实施第一遍 messages.ts 写到 702 行超 B3-I5 600 上限。`rowsToTurns` + `safeParseJsonArray` + `previewFromContent` 123 行抽到 `gui/src/stores/messages/rowsToTurns.ts` 子文件 pure-function module。messages.ts 落 584 行 ✓。
7. **`patchMessages` + `fireSessionMirror` 内部 helper** —— 替代 useAppStore.applyRuntimeUpdate。`patchMessages` 是 set-time pure 数据 mapping；`fireSessionMirror` 在 set 之后单独调 sessionsStore.applyDerivedFromRuntime。两步分开保证 Zustand 单 set 调用 + 跨 store 写在 set 之外。
8. **Rust 端零改动**（B3-I4 守）—— sub-plan §1 拒绝 T5.2 (`messages-appended` event) 和 T5.3 (16ms batch)，原因是 N7 perf 实测 1.42 ev/s React 端无压力 + B3-I4 不动 Rust 端规则。实施实测 Rust 126/126 测试 unchanged。
9. **历史 doc comment 接受残留**（V3 grep verification）—— `_runtimes` / `applyRuntimeUpdate` 在 messages.ts:56 / runtime.ts:21 各留 1 处叙事性历史引用（"fields that lived on `useAppStore._runtimes` pre-B3 M5"），不影响代码语义。policy 是 V3 grep 不允许的是符号 reference，叙事引用允许。
10. **EMPTY constants 类型决策** —— 第一遍写成 `readonly Turn[]` 立刻撞 MainView prop type `Turn[]` 不兼容（TS2352）。改成 mutable type + Object.freeze 双层 = component 借用零阻力 + runtime safety 仍在。`readonly` 修饰符在 product code 是装饰品，freeze 才是 enforcement。

## Rejected alternatives

- **拆 M5a (skeleton only) + M5b (full migrate)** —— M5a 不动 useAppStore.\_runtimes，messagesStore 全 dead code 等 M5b。0 行为变化 = 浪费 commit slot。（sub-plan §8 Reject #1）
- **拆 M5a (3 字段 onClose 先迁) + M5b (剩余 5 字段)** —— 违反 B3-I3 "同 capability 不并存"，dogfood 期间双 store 状态分裂难 debug。（sub-plan §8 Reject #2）
- **Rust 端引入 `messages-appended` event + 16ms batch** —— 违反 B3-I4 "B3 内不动 Rust 端 (除了加 emit 事件的 minor patch)"。messages-appended 是 spawn_emit_task 内累积 + flush 新逻辑，不是 thin emit。N7 perf 实测 1.42 ev/s React 不构成压力。推 B4 supervisor push 配合一起做。（sub-plan §8 Reject #3）
- **`activateSession` 留 useAppStore** —— Option A 把 orchestrator 留在 useAppStore 是「不动既有结构」的诱惑。M6 收尾时还得搬一次，等于多一个 commit。Option B 一次搬到位。（sub-plan §8 Reject #4）
- **React Context 替代 useMessagesStore** —— O1 已 resolved 沿用 Zustand。Context 跨 slice 协调成本更高 + strict-mode 兼容性需要 store-side enrichment + 不能 store getState() 同步读。（sub-plan §8 Reject #5）
- **`userSubmitTick` 迁 uiStore** —— mapping doc O8 提议过。re-evaluate：3 个 messages action 写它，迁 uiStore 要跨 store 调 uiStore.bumpSubmitTick = 不必要的 indirection。global 留 messagesStore 字段顶层（不进 byId）。（sub-plan §8 Reject #6）
- **EMPTY 类型保留 `readonly Turn[]`** —— 强行 type-safe 但 component prop 不接受。改成 mutable + frozen runtime 才两全。
- **保留 lib/sessions.ts deriveSessionStatus 接 `PerSessionMessages` 直接 import** —— 会让 lib 反向依赖 store 模块。改 structural view (`MessagesView` interface)，lib 不知道 store 存在。
- **`clearSessionRuntime` 保留 dynamic import** —— M4b 时是必要的（防止双向 module-eval cycle，messages 还没出生）。M5 后 runtime.ts / messages.ts / sessions.ts 已 cycle 共存，dynamic import 反而 misleading。改 static 跟 pattern 一致。

## Open questions

- **Dogfood T5.16 7 cluster 全过** —— V1 (typecheck) ✓ / V2 (Rust unchanged) ✓ / V3 (grep no leakage) ✓ / V4 (cross-store stub 清完) ✓；剩 V5 (auto-scroll regression) / V6 (turnIndexOffset PK correctness) / V7 (cross-store mirror correctness) + 7 scenario cluster (basic / approval / ask_user / /btw / streaming auto-scroll / session restore / bridge crash) 要 JC 跑 dev mode 真验。**风险：R3 + R4 高 severity**（R3 auto-scroll effect deps 改变 / R4 turnIndexOffset PK 防碰 SQLite restore 路径）。
- **`useSubmitTick` global vs uiStore re-eval** —— 当前实施保 messages global 字段。若 dogfood 发现 MainView scroll trigger 在 session 切换时 misfire（不应该 misfire 因为 tick 现在不跟 sessionId 关联），考虑迁 uiStore。
- **M6 prefsStore scope 重定** —— 现 useAppStore.ts 465 行，90% 是 prefs lifecycle (gaConfig / approvalConfig / yoloMode / yoloIntroSeen / conversationWidth)，剩下 hydrateFromDB orchestrator 约 100 行 + 4 个 prefs setter 约 100 行 + seedMockSessions forward。M6 可以彻底删掉 useAppStore.ts (M6 T6.6 推荐) 或留 hydrate 一个 thin file。M6 sub-plan 写作时再定。
- **Streaming perf re-measure post-M5** —— N7 baseline 是 useAppStore.applyRuntimeUpdate 一次 set 同时更新 _runtimes + projection mirror + sessions row。M5 后 messagesStore.appendInFlightDelta 单字段 set + fireSessionMirror 跨 store 写。两种实现 React re-render count 应该相当（一次 set 触发一次 frame），但实际测一遍才有 confidence。Dogfood T5.16 期间跑 P1/P2 即可。
- **sessions.ts 1054 行 + messages.ts 584 行的子文件拆分时机** —— sessions.ts 早超 B3-I5 600 上限，messages.ts 单文件守住但 `messages/rowsToTurns.ts` 子文件已建立 pattern。M6 后 sessions.ts 是否同步分子文件 = 单独 cosmetic commit 决策。
- **lib/sessions.ts MessagesView 抽到 types** —— 当前定义在 lib/sessions.ts 内部，跟 deriveSessionStatus 同文件。如果未来 prefsStore / 其它 slice 也需要 mirror status，抽到 `types/session.ts` 或 `types/messages.ts` 是合理一步。M6 实施时 re-eval。
