## Context

下午把 PRD v0.3 + Yole Core 重构 playbook 推上去之后，开始走 v0.1 公开发布流程。原本 SOP（[release-workflow.md](../release-workflow.md)）是这样的：dry-run 验 CI → tag → CI 自动 build 三平台 + 创建 draft → review + publish。

实际跑下来踩了几个坑，决策叙事 + 修正 SOP 记到这里。同时把今天关于 macOS / Mac 写作用词的细节也定下来。

## Decisions

### D1. Tiered release：macOS RC + Windows Alpha 分两个 release

JC 提出：Mac 我本地 smoke 跑过（x64），算 RC 阶段；Windows 我没机器验证但 GA 社区有人愿意 dogfood，算 Alpha 阶段。两边不应该绑成一个 release。

最终结构：
- **v0.1.0-rc.1**：macOS only（aarch64.dmg + x64.dmg），prerelease=false 标 Latest
- **v0.1.0-alpha.1**：Windows only（x64-setup.exe），prerelease=true

两个独立 release 各自 release notes，互相 link。这给将来"信心不齐的平台一起出"提供了模板——把 SOP 写下来（详 release-workflow.md 新增 "Tiered release strategy" 节）。

### D2. CI macos-13 Intel queue stalled → 手动 fallback

跑 v0.1.0-rc.1 release.yml 时：
- ✅ macos-14 aarch64：~10 min build 完，artifact 10.5 MB
- ✅ windows-latest x64：~10 min build 完，artifact 5.5 MB
- ⏳ **macos-13 x64 (Intel)：长时间 queued**——SOP §"CI 故障排查 macos-13 runner 排不到" 说 5-10 min 通常自然解决，实际等了几十分钟没动静

`release` job 依赖三平台都过，Intel 卡住 = draft Release 永远不出。

**JC pragmatic call**："我本地已经 build 好 x64.dmg 自己测过了，能不能先发布?" 答案是可以——v0.1 是 macOS dogfood，本地 x64 + CI aarch64 已经覆盖了 Mac 全 arch。

**手动 fallback 路径**（今天踩出来的，必须写进 SOP）：

```bash
# 1. 取消卡住的 CI run（释放 slot + 清状态）
gh run cancel <stuck-run-id>

# 2. 从已经 success 的 CI build 下 artifact
gh run download <run-id> -n yole-macos-14-aarch64  # aarch64 dmg
# x64 用本地 pnpm tauri build 产物

# 3. 手动 gh release create
gh release create v0.1.0-rc.1 \
  --draft --prerelease \
  --title "Yole v0.1.0-rc.1 · macOS (Release Candidate)" \
  --notes-file /tmp/yole-rc1-notes.md \
  Yole_0.1.0_aarch64.dmg Yole_0.1.0_x64.dmg

# 4. review draft → publish
gh release edit v0.1.0-rc.1 --draft=false
```

这个 fallback 不光救今天，**Intel runner 长期不稳**（SOP §"Intel runner deprecation 应对"已写过 deprecation 路径），手动 fallback 提前内化为常规预案而非应急。

### D3. GitHub 不允许 prerelease 标 Latest——Mac RC 摘掉 prerelease flag

publish 完两个 prerelease 之后，JC 反馈 GitHub 主页右侧 sidebar "Releases" widget 显示 "2 tags · Create a new release"，看起来像没发版。

原因：GitHub sidebar widget 默认只 promote "Latest" release，而 GitHub API 不允许 prerelease 标 Latest（实测 422 Validation Failed "Latest release cannot be draft or prerelease"）。

**解决**：`gh release edit v0.1.0-rc.1 --prerelease=false --latest`——把 Mac RC 的 prerelease flag 摘掉，标 Latest。

代价：Mac RC 失去 GitHub 层 "Pre-release" 灰色 badge。

**保留的诚实**：
- Release title 文字保留 "(Release Candidate)" —— 用户看 title 仍然知道 quality 等级
- Release notes 内部完整解释 RC quality（aarch64 没 smoke / 帮 dogfood 意义最大 / 已知限制等）
- Win Alpha 保留 prerelease flag——它确实更早期

Trade-off 接受：v0.x 阶段开源项目大多数都这么干，dogfood 期没 stable release 但需要 visibility。

### D4. macOS vs Mac 用词约定

JC 问：用词上 Mac 还是 macOS 更好？

定下来按场景分：

| 场景 | 用词 | 例 |
|---|---|---|
| OS / 平台名 | `macOS` | "Requires macOS 14"、"macOS app"、release title "Yole v0.x · macOS (RC)" |
| 设备 / 硬件 family / 用户 | `Mac` | "Apple Silicon Mac"、"Intel Mac"、"your Mac"、"M1+ 用户" |

Apple 自己的惯例：macOS = OS，Mac = 设备 family。主流 SaaS（Slack / Notion / Figma）下载页全用 macOS。

**今天落地**：
- Release title v0.1.0-rc.1 从 "Mac (Release Candidate)" 改 "macOS (Release Candidate)"
- 两份 release notes 内部"Mac RC"、"跟 Mac 体验"等改"macOS"
- README badges 已经是 macOS（保留）
- README install section header `### macOS` 已经对（保留）
- "Intel Mac" / "Apple Silicon Mac" 等设备 reference 保留 Mac

**写进 CLAUDE.md** "Platform terminology" section，未来一致使用。

### D5. Announcement drafts 在 /tmp/，不入库

通告草稿（GA 群中文 + Twitter 英文短 + HN/Reddit 英文长）写到 `/tmp/yole-announce-{zh,en}.md`，**不 commit 进仓库**。理由：

- 通告是发出去的临时内容，不是 reusable doc
- 每次 release 内容不同（具体功能 / 已知限制 / 安全要点）
- **但模板结构可复用**——所以把模板（标题 / 段落顺序 / 必备字段）写进 release-workflow.md 的 "Announcement templates" section

下一次发版时按 release-workflow.md 模板填具体内容写新 /tmp/ 文件，发完丢。

### D6. v0.5 路线图提到通告里——commit to externals

中文通告末尾提了 v0.5 Yole CLI / Supervisor Agent。这是一个 push back 点（我提过："你 commit 这条 message 后就有外部期待了"），JC 接受。

理由（事后整理）：
- v0.5 路线图 PRD + refactor playbook 已经 ship 到 main，公开
- README 也已经有 v0.5 章节
- 通告里提一句不是新增承诺，是已有承诺的复述

但需要内化的事：**v0.5 任何延期 / scope 缩减 / 取消 → 必须配 devlog + 通告解释**。不能"悄悄不做"——已经公开 commit 了。

### D7. 公开 announcement channel：仅 GA 社区群，Twitter/HN 不发

JC 决定先只发 GA 用户群中文通告。Twitter 英文短版 + HN/Reddit 英文长版 draft 保留但不用——国际化曝光不是 v0.1 重点。

将来如果想做国际化曝光，模板已经在 /tmp/，照着改即可。

## Rejected alternatives

### 等 Intel runner 排队到底

讨论过：再等 30 min / 1 小时 / 数小时直到 Intel 排到。否决：

- SOP §"Intel runner deprecation 应对"已经标 Intel runner deprecated，不知道何时下线，长期不可靠
- 今天等可能等到，但下次还要等，**不是单次 bug 是结构问题**
- 本地 build 完全 cover x64，CI Intel 只是同一个 .dmg 的不同来源

接受 manual fallback 作为 SOP 一部分。

### 单一 v0.1.0 release 包 3 个平台 artifact

讨论过：把 Mac aarch64 + Mac x64 + Win x64-setup.exe 都塞进一个 v0.1.0 release，notes 内部说明 tier 差异。

否决：JC 明确想分 tier framing——macOS RC 信心更高，Windows Alpha 信心更低。**Tier 信号塞在 notes 里容易被用户忽略**，分两个 release 视觉上 hard separation 更清楚。Release 列表上用户看到 "macOS RC (Latest)" + "Windows Alpha (Pre-release)" 两条，一眼就懂。

代价：管理两个 release 而不是一个。可接受。

### 保留 Mac RC 的 prerelease flag，接受 sidebar widget 空着

讨论过：不去管 sidebar widget，让"2 tags · Create a new release" 显示。

否决：JC 反馈 sidebar 空看起来像没发版——影响访客第一印象（"这项目活着吗"）。摘掉 prerelease flag 换来 sidebar visibility 值得。

### 把通告写成可复用 template 文件入库

讨论过：把 `/tmp/yole-announce-zh.md` 直接 commit 成 `docs/release-announcement-template.md`。

否决：今天的通告内容（具体 LLM 列表 / 已知限制 / 安装命令）跟 v0.1 强绑定，下次 release 都不一样。模板结构（标题 / 段落 / 必备字段）才可复用——这个写进 release-workflow.md 的 "Announcement templates" section。

### 在 CLAUDE.md "Brand wordmark rules" 里加 macOS/Mac 规则

讨论过：把 macOS/Mac 加到 Brand wordmark rules section（同样是写作约定）。

否决：Brand wordmark rules 是 Yole 自己的品牌名（Yole / YOLE / GenericAgent / GA）的呈现规则，跟平台名是不同概念，强行揉一起会让 section 主题模糊。**单独开 "Platform terminology" section** 更清楚。

## Open questions

### O1. Intel runner 真下线之后怎么办

[release-workflow.md §"Intel runner deprecation 应对"]已经列了 3 个方案（drop Intel / 本地 build / Apple Silicon 跨编译）。今天验证了"本地 build + manual fallback"路径可行，方案 B 实操化。

但**长期可持续性问题没解**：JC 永远要在自己机器上手动跑 build？v0.6+ 真有 Win 用户 + Intel Mac 用户后，手动每次都不轻松。

候选解：方案 C（Apple Silicon 跨编译 x86_64-apple-darwin）。等 Intel 真下线时再 evaluate。

### O2. 何时第一个非 prerelease release？

今天 Mac RC 摘掉 prerelease flag 是 hack——title 仍然叫 "(Release Candidate)" 但 GitHub 层不再是 prerelease。

**真正的"production release" 何时出？** 当：
- aarch64 也有真实用户 smoke 反馈
- Windows 经过 Alpha 期社区 dogfood
- 至少 1 个非 JC 的 dogfooder 长期使用
- 关键 bug 收口

那时打 `v0.1.0`（无 -rc 后缀）正式 release，title 改成 "v0.1.0 · 第一个稳定版"。Mac RC 可以删或保留作历史。

时间：估计 v0.1.0-rc.1 公开 dogfood **2-4 周**后，看反馈。

### O3. v0.5 Yole Core ship 时 release 流程会变吗？

v0.5 引入 Yole CLI = 增加一个 deliverable（CLI binary）。release flow 需要：
- bundle CLI binary 进 .app（[B4 playbook](../refactor/B4-cli-bg-artifact.md) 已规划 sidecar）
- discovery file 写入逻辑
- 新增 `yole` CLI test step in CI
- Release notes 加 CLI usage 一节

这部分等 B4 接近时再 audit SOP。**今天的 SOP 不需要为 v0.5 提前改**。

### O4. Code signing 啥时上？

[release-workflow.md §"未来工作 (v0.6+)"]已写：
- Mac Apple Developer Program $99/年
- Windows EV cert $200-400/年

判断节点："dogfood 群体之外有真用户量时"。

今天 v0.1 ship 之后到 release 反馈到来这段时间，可以观察 GA 社区的 download 数 / Issues 数。如果 1 个月内 100+ downloads + 多个非 JC 的 issue 报告者，**v0.6 启动 code signing 流程**。
