## Context

GA upstream 2026-05-13 出的 `tuiapp_v2.py`（终端版多 session UI）带了三个我们没规划过的命令——`/btw` 侧问、`/branch` 分支、`/rewind` 悔棋。审完源码后确认这三件是**正交于** Approval / Multi-session / Restore 的新交互原语，不是我们既有能力的重复。

GA TUI 的实现路径是 monkey-patch `GenericAgentHandler._handle_slash_cmd`（[btw_cmd.py:128](../../../GenericAgent/frontends/btw_cmd.py:128)）+ 直接 deepcopy/mutate `backend.history`（[tuiapp_v2.py:947, :961](../../../GenericAgent/frontends/tuiapp_v2.py:947)）。我们做不了——违反项目宪法第 4 条「不 monkey-patch GA 工具实现」。

本文档定义三个命令在 Yole 架构下的等价实现：bridge IPC 命令 / 事件 + desktop UI 草案。

## Decisions

### 一、`/btw` Side Question

**Rationale** — 用户在主 agent 长任务运行中想顺嘴问个事（"它现在在搞什么？"/ "刚才那个文件路径是什么来着？"）。现有选项：(a) 等任务完成再问，(b) `/stop` 后问再重启，(c) 开新 session 丢上下文。三个都差。`/btw` 给第四个选项：**主 agent 不动，跑一个 sub-agent 复用当前历史做单次纯文本作答**。

#### IPC 协议增量

**新增 command `side_question`**

```json
{
  "kind": "side_question",
  "question": "你刚才读到的 agentmain.py 里有几个 class？",
  "id": "sq_abc123"
}
```

字段：
- `question`：用户的问题，≤2000 字符
- `id`：desktop 生成，用于关联 response

**新增 event `side_question_response`**

```json
{
  "kind": "side_question_response",
  "sessionId": "sess_xxx",
  "id": "sq_abc123",
  "status": "success",
  "answer": "（sub-agent 的纯文本回答）",
  "elapsedMs": 3450,
  "timestamp": "..."
}
```

字段：
- `status`：`"success" | "timeout" | "error"`
- `answer`：sub-agent 的纯文本回答；status=error 时为错误描述
- `elapsedMs`：耗时

**约束**：
- `side_question` 可以在 agent **任意状态**（idle / running / waiting_approval）下发出——这是 `/btw` 的核心卖点
- bridge 收到后**不抢占主 agent**：起独立 worker thread，在 `backend.lock` 下 deepcopy `backend.history` 后释放锁，调 `backend.raw_ask` 单次拉答
- bridge 永远不写 `backend.history`
- 超时硬上限 120s（与 GA TUI 一致）

#### Bridge 行为

复用 GA `frontends/btw_cmd.py` 的 `handle_frontend_command(agent, query)` 同步接口（不走 monkey-patch 的 `install(cls)` 那条路）。bridge 里：

```python
# bridge/handlers.py
def handle_side_question(self, question: str, sq_id: str) -> None:
    def worker():
        started = time.time()
        try:
            answer = btw_handle_frontend(self.agent, f"/btw {question}")
            status, elapsed = "success", time.time() - started
        except Exception as e:
            answer = f"{type(e).__name__}: {e}"
            status, elapsed = "error", time.time() - started
        self.emit(SideQuestionResponseEvent(
            id=sq_id, status=status, answer=answer,
            elapsed_ms=int(elapsed * 1000),
        ))
    threading.Thread(target=worker, daemon=True, name=f"sq-{sq_id}").start()
```

注意：调用 `frontends.btw_cmd.handle_frontend_command`，不要走它的 `install(cls)` monkey-patch 入口。前者是纯函数，后者改 GA 内部状态。

#### Desktop UI

**触发方式**（两条互不冲突）：
1. **Composer slash 命令**：用户输 `/btw <question>` 后 Enter，desktop 拦截，不走 user_message
2. **Composer 旁的 "Ask aside" 按钮**：仅在 agent running 时显示；点击弹一个轻量 popover 让用户输入，Enter 提交

**渲染**：在主对话流中插入一对**黄色侧边条**的特殊气泡：

```
┃ 🟡 你（侧问）：刚才读到的 agentmain.py 里有几个 class？
┃
┃ 🟡 旁问回答（3.5s）：3 个 class：GenericAgent、BaseHandler、ChainedHandler。
```

视觉规范：
- 左侧黄色 1px border（`--color-warning` 或 `#d4a73c`）区分主流
- 标签文案：`你（侧问）` / `旁问回答`，用户/AI 角色清晰
- 不出现在 Tool Timeline、不触发 Approval Dock
- 不计入 Turn 编号（"第 N 步" 跳过它）

**持久化**：写入 `events` 表，新增 event kind `side_question` + `side_question_response`。`replayHistoryToBridge` **跳过**这两类事件——主 agent 历史里没它们，恢复时也不该有。

#### 边界情况

- agent 正在跑、`/btw` 一发再发：每个独立 worker，bridge 不去重——用户想连问就让他连问
- bridge 在 `/btw` 还没回时收到 `shutdown`：worker 是 daemon thread，进程退出自然死，丢弃未完成的 side question
- LLM 失败：worker 捕获异常 → `status: "error"`；UI 显示红色气泡 + Retry 按钮

### 二、`/branch` Fork Session

**Rationale** — 用户跑了 5 步，发现想试两条不同走向（"如果让它用 React 19 而不是 React 18 重写呢？"）。现在只能开新 session 丢上下文。`/branch` 复制当前 history 到新 session，原 session 不动，给"留底回退"的安全感。

#### IPC 协议增量

**零新增 IPC**。Branch = `spawn 新 bridge + load_history(snapshot of source)`，全部用现有协议。

#### 实现路径

Desktop 侧做的事：
1. 用户右键 sidebar 选 "Branch this session"（仅 source idle 时启用）
2. 弹 dialog 让用户改名（默认 `{source.title} · 分支`）+ 选目标 project（默认继承 source.projectId）
3. 从 SQLite 读出 source session 的所有 turns，序列化为 `load_history.messages`
4. 创建新 session row（继承 LLM index、project、cwd）
5. spawn 新 bridge → 等 `ready` → 发 `load_history` → 等 `history_loaded` → 切到新 session

#### 约束

- **source 必须 idle**：running / waiting_approval 时禁用菜单项；如果用户绕过（理论上不会有路径），desktop 端校验拦下
- **当前 load_history 的 lossy 行为**：[ipc-protocol.md §10 Open Items](../ipc-protocol.md#10-open-items) 已记录——只在 `NativeClaudeSession` 下保证 text 恢复，tool_use / tool_result blocks 会丢。Branch 沿用此限制：分支 session 知道**对话脉络**但不知道**具体 tool 调用细节**。可接受——这跟 Session Restore 的语义一致

#### Desktop UI

Sidebar 右键菜单 SessionRow 增加一项：

```
Open in new window         ⌘↩
Move to project          ▸
Branch this session
────────────────────
Archive                    ⌫
```

Branch 项的 `disabled` 条件：source.status !== "idle"。disabled 时 tooltip：「运行中无法分支，等任务完成或 /stop 后再来」。

Branch dialog 复用 CreateSessionDialog 的视觉模板，仅替换 header 文案为「分支会话」+ 预填 title。

**Bridge 真跑通后**：新 session 顶部插一条特殊 system message：「这个会话从 #{source.id} 分支而来（{N} 步）」，给用户回溯入口。点击跳回 source。

#### 边界情况

- source session 突然被 archive：分支 dialog 关闭时再读一次 source 状态，已 archive 就报错退出（不允许从 archived 分支）
- 多次连续 branch：每个 branch 独立，互不影响。新 branch 也可以再 branch
- LLM 切换历史中的 sub-history：GA `next_llm` 内部会把 `backend.history` 迁到新 client。如果 source 期间切过 LLM，branch 时拿的是最新 backend 的 history（这是对的——LLM 切换不丢上下文）

### 三、`/rewind <n>` Undo Turns

**Rationale** — 用户跑到第 8 步发现"诶第 5 步那个工具调错路径了，后面三步都白跑"。现在只能重开 session 重头来。`/rewind 3` = "回到第 5 步前那个状态，从那里重来"——比 `/branch` 轻、比"重开"省时。

#### IPC 协议增量

**新增 command `rewind`**

```json
{
  "kind": "rewind",
  "turns": 3
}
```

字段：
- `turns`：要回退的 user-turn 数量，≥1

**新增 event `rewound`**

```json
{
  "kind": "rewound",
  "sessionId": "sess_xxx",
  "turnsRemoved": 3,
  "historyEntriesRemoved": 8,
  "remainingTurnCount": 5,
  "timestamp": "..."
}
```

字段：
- `turnsRemoved`：实际回退掉的 user turn 数（≤ 请求值，如请求 5 但只能回 3 则返回 3）
- `historyEntriesRemoved`：`backend.history` 实际删了几个 entry（user msg + assistant msg + 各种 tool blocks 都算）
- `remainingTurnCount`：剩余的 rewindable user turns 数（让 UI 决定 disable 状态）

**约束**：
- agent 必须 idle 才允许 rewind（running / waiting_approval 时 emit `error`）
- `turns` 越界（超过实际可回退数）→ emit `error` 不执行

#### Bridge 行为

参考 `tuiapp_v2.py:_cmd_rewind` 的扫描逻辑：

```python
def handle_rewind(self, n: int) -> None:
    if self.session_state.status != "idle":
        return self.emit_error("Cannot rewind while running. Abort first.")
    history = self.agent.llmclient.backend.history
    # 仅 user-role 且非 tool_result-only 的 entry 才算一个可回退的 turn
    rewindable_indices = []
    for i, m in enumerate(history):
        if m.get("role") != "user":
            continue
        c = m.get("content")
        if isinstance(c, list) and all(
            b.get("type") == "tool_result" for b in c if isinstance(b, dict)
        ):
            continue
        rewindable_indices.append(i)
    if n < 1 or n > len(rewindable_indices):
        return self.emit_error(f"Invalid rewind count: 1-{len(rewindable_indices)}")
    cut = rewindable_indices[-n]
    removed = len(history) - cut
    history[:] = history[:cut]
    self.emit(RewoundEvent(
        turns_removed=n,
        history_entries_removed=removed,
        remaining_turn_count=len(rewindable_indices) - n,
    ))
```

注意：直接 `history[:] = history[:cut]` 是在 bridge 子进程里操作 GA 内部状态。这**不**违反 non-invasive——non-invasive 禁的是 monkey-patch GA 工具实现，读写 `backend.history` 是 PRD §4 明确许可的接入方式（"读取 / 注入 backend.history"）。

#### Desktop 持久化策略：非破坏性

不删 SQLite 的 `agent_turns` rows。新增 migration 005：

```sql
ALTER TABLE agent_turns ADD COLUMN rewound BOOLEAN NOT NULL DEFAULT 0;
ALTER TABLE agent_turns ADD COLUMN rewound_at TIMESTAMP NULL;
```

收到 `rewound` 事件后：把 source session 中 `turn_index > cutoff` 的 rows 标 `rewound=true`，主对话流默认隐藏。Sidebar 状态栏显示 "已回退 N 步"，提供 "查看已回退" 按钮可临时展开（灰显、不可交互）。

**为何不删**：(a) 审计可追溯（"那次回退掉的是什么"）；(b) 心理安全感（"万一回错了能撤回"）—— V0.3 可考虑加 `/unrewind` 让用户恢复（需要同时反向操作 backend.history，复杂度高，先不做）。

#### Desktop UI

**触发方式**：

1. **Composer slash 命令**：`/rewind` 列出可回退的 turn（带 1-10 序号 + preview）；`/rewind N` 直接确认
2. **Conversation message action**：每条 user message 的悬停 actions 加 "↶ Rewind to here"——更直观，一键回到这条消息**之前**的状态

**确认对话框**：

```
回退 3 步？

将隐藏这 3 步：
  · #6 帮我重写成 React 19 风格
  · #7 顺便加上 ErrorBoundary
  · #8 用 jest 不用 vitest

GA 会回到 #5 完成时的状态。已回退的内容不会消失，可在
"已回退" 视图查看。

  [取消]  [回退 3 步]
```

destructive 视觉 treatment（红色 confirm button），与 Archive / Delete project 同源。

#### 边界情况

- 用户连续 rewind：每次都基于**当前** rewindable turns。`rewound=true` 的 row 不参与计数
- bridge 进程崩溃：`rewound` 事件已发到 SQLite 持久化，重启 bridge 走 Session Restore 会 replay 已存的 history（不含被 rewound 的 turn）→ 自然恢复回退状态
- rewind 后立刻 `/btw`：side question 拿的是 deepcopy 后的 `backend.history`——已经是回退后的状态，符合预期

## Rejected Alternatives

### A. monkey-patch `_handle_slash_cmd` 复用 GA TUI 路径

GA TUI 的 `btw_cmd.install(cls)` 直接 patch `GenericAgentHandler._handle_slash_cmd`。

**否**：违反 CLAUDE.md 项目宪法第 4 条「不 monkey-patch agent_runner_loop 或 do_* 工具实现」。`_handle_slash_cmd` 是 handler 链路的核心入口，monkey-patch 它就是改 GA 内部行为。我们用 IPC command 路径——desktop 拦截 slash 后走专用 IPC，跟 user_message 走的 dispatch 路径**完全分离**。

### B. desktop 端直接操作 SQLite + 假装 rewind 了

不动 bridge，desktop 只删/隐藏 turns，下次 user_message 时让 bridge 走 Session Restore 用截短的 history 重启。

**否**：违反 "bridge is truth, desktop is view"。bridge 进程里 `backend.history` 还是完整的，desktop 视图跟 bridge 实际状态不一致——下一个 user_message 一发，GA 用的还是老 history，跟用户看到的"已回退"对不上。要么真的 mutate bridge，要么 respawn bridge——前者轻量、后者破坏 multi-session LRU。选前者。

### C. `/rewind` 用 destructive 删除 SQLite rows

**否**：(a) 失去审计轨迹；(b) 没法做 V0.3 的 unrewind；(c) "Yole 是 GA 之上的可观察 / 可审计层" 是核心定位之一，不该破坏自己的 audit log。非破坏性 marker 多一个 column，工程成本可忽略。

### D. `/branch` 在同一 bridge 进程里 fork agent

GA TUI 因为是 in-process 多 agent，能直接 `copy.deepcopy(backend.history)` 进新 agent。我们 bridge-per-session 做不到。

**否**：跟我们 multi-session 隔离原则对立。branch 走 "spawn 新 bridge + load_history" 路径——稍微重一点（多一次 GA 启动 ~2-3s），但保持架构一致。用户感知上 branch 立刻出现在 sidebar（pending 状态），bridge 后台 spawn 不阻塞 UI。

### E. ChoiceList for LLM 切换

GA TUI `/llm` 无参数时弹 inline ChoiceList。

**否**：我们 Composer 已有常驻的 LLMPill + Radix Popover（[Composer.tsx:222](../../desktop/src/components/conversation/Composer.tsx:222)）。这是更好的 pattern——always-visible 显示当前模型、不需要先打命令、不污染对话流。ChoiceList pattern 本身有价值（见 Open Questions 第 3 条 ask_user），但不是给 LLM 切换用。

### F. `/btw` 允许使用工具（restricted mode）

让 side agent 有 read-only 工具（file_read / web_scan）。

**否**（暂时）：GA TUI 显式禁止 sub-agent 用工具（"You have NO tools"）。这是好的边界——一旦给工具，side question 就可能跑成嵌套任务，复杂度爆炸。V0.2 先按 "一次纯文本回答" 跑通，V0.3+ 再看是否要 restricted-tool 模式。

## Open Questions

1. **`/btw` 多轮 vs 一次性**：当前设计是一问一答，回答之后 thread 结束。如果用户对 sub-agent 的回答还想追问呢？两种走法：(a) 强制再发 `/btw <followup>` 起新一个 sub-agent，每次都新 deepcopy；(b) 给 side question 一个临时 history，UI 上能"继续侧问"。倾向 (a)——简单，符合"sub-agent 是一次性轻量进程"心智。

2. **`/branch` 名字交互**：默认 `{source.title} · 分支` 让 sidebar 一眼能看出血缘，但长会话标题会被截断。是否需要单独的 "Branch from" 元数据字段在 SessionRow 上显示（小标签 / icon）？倾向加 `branched_from_session_id` column + sidebar 上一个 fork icon。

3. **ChoiceList 用在哪——`ask_user` 才是它的归属**：我们当前 [ipc-handlers.ts:253](../../desktop/src/lib/ipc-handlers.ts:253) `ask_user` 事件只 `console.debug`，没渲染！这是 V0.1 漏掉的功能（PRD §11 提过 ask_user 工具，但 UI 没实现）。GA TUI 的 inline ChoiceList 在这里恰好——主对话流里插一条 system message + OptionList，用户选完一项后塌缩成 `✓ {选项}` 消息 + 自动发 `ask_user_response` IPC。这是 V0.2 应该一起做的事。**问**：要不要把 `ask_user` UI 实现也并入 V0.2 这一批？

4. **`/rewind` 与 Edit Message 的冲突**：我们 V0.1 已有 "Edit user message" 功能（用户改了一条历史用户消息重发）。`/rewind N` 隐藏了某些 turns 后再 edit 更早的消息会怎样？逻辑上 edit = rewind 到那条 + 用新内容重发。是否要把 Edit 内部实现重构为 "rewind + user_message"，让两者共享语义？倾向重构——单一真相更清晰，但有 backwards compat 风险（V0.1 用户的 edit 历史）。需要工程评估。

5. **`load_history` 富 adapter（Open Items 遗留）**：[ipc-protocol.md §10](../ipc-protocol.md#10) 已记录 tool_use/tool_result blocks 在 load_history 中丢失。`/branch` 直接踩这个限制——分支的 session 不知道 source 的 tool 调用历史。是否要先把 adapter 升级了再 ship `/branch`？倾向**不要**：先 ship lossy 版本拿用户反馈，看是不是真痛点。Adapter 升级是个独立工程（每个 backend 一个），单独排期。
