## Decisions

### Sub-plan ship 锁定拆分策略（commit `0943c91`）

[B3-M4-sub-plan.md](../refactor/B3-M4-sub-plan.md) 348 行落地，mirror M3 sub-plan 结构：

1. **拆 M4a + M4b 不单 commit**：M4a 单 Rust（trait + tests，frontend 不动 → 行为 byte-identical → dogfood gate 颗粒度极简）+ M4b 单 frontend（sessionsStore 抽离 + 22 call site swap + lib/db.ts 写路径全删）。中间 dogfood 1 天 + 风险敞口约束在「Rust 编译 + cargo test」内。
2. **playbook 「18 trait method」改 16**：12 session method + 4 project method = 16。`setActiveSession` / `activateSession` / `setActiveProjectFilter` 是 pure display state operation **不需要 trait method**——`setActiveSession` 是 in-memory pointer，`activateSession` 是 frontend orchestrator（compose setActive + restoreTurns + spawnBridge），`setActiveProjectFilter` 是 UI filter mode 不持久化。
3. **Origin 参数策略**：所有用户/agent 可触发的写都要 `Origin`（B2 M5 contract）。系统内部行为（bump / clear_unread / set_session_llm）不要——frontend 不传，CLI 也碰不到（B4 才暴露 CLI）。
4. **Bulk method first-class trait surface**：单 trait method 循环调用 N 次 over Tauri IPC 不可接受（GUI EarlierDialog 一次批量 archive 30 session 走 30 trip 肉眼可感卡顿）。bulk method 跟 single method 等位。
5. **R1-R8 risk register + 7 verification gate**：R2 (persistSession 残留 caller silent failure) / R6 (double-Option `Option<Option<String>>` 跟 frontend `partial` shape 不完美对齐) / R7 (frontend-assigned id 冲突如何处理) 写在前面避免实施时纠结。

### M4a Rust trait + tests ship（commit `eca7d65`）

**新增 surface（16 trait method）**：
- Sessions (12): `create_session` / `archive_session` / `unarchive_session` / `rename_session` / `set_session_pinned` / `delete_session` / `assign_session_to_project` / `bump_session_after_turn` / `clear_session_unread` / `bulk_archive_sessions` / `bulk_unarchive_sessions` / `bulk_delete_sessions`
- Projects (4): `list_projects` / `create_project` / `update_project` / `delete_project`

**Input types (camelCase wire)**:
- `CreateSessionInput { id, title, projectId?, selectedLlmIndex?, selectedLlmDisplayName? }` —— id caller-assigned, conflicts → `invalid_args`
- `CreateProjectInput { id, name, rootPath?, icon?, color? }`
- `ProjectPatch { name?, rootPath??, icon??, color??, pinned? }` —— `??` double-Option for nullable string columns（`Some(None)` clears to SQL NULL）

**Error mapping**:
- 不存在 id → `NotFound` (exit 3)
- 空 title / id 冲突 / archived session pin reject / FK violation → `InvalidArgs` (exit 2)
- DB unavailable → `DbUnavailable` (exit 4)
- 其它 → `Internal` (exit 1)

**`set_session_pinned` archived reject**：UPDATE 前先 SELECT status，archived row 主动返回 `InvalidArgs` 而不是 silently no-op，让 caller 拿到清晰错误类别。

**`unarchive_session` 不存在 row 设计**：UPDATE 0 rows_affected 时不能区分「missing」vs「already idle」，所以**先 UPDATE 再 session_brief()** —— brief 失败时报 NotFound，brief 成功但没变化时返回当前状态（idempotent no-op）。

**`map_constraint_err` shim**：sqlx `Database` error 没有 Rust-level discriminator，看 SQLite 错误消息字符串包含 `foreign key` / `unique` / `check` / `primary key` 时手动映射成 `InvalidArgs`。pure-Rust 没有 better way（升级 sqlx 0.9 改善 typed error）。

**`truncate_summary` + GUI `truncateSummary` 必须语义一致**：Rust 端 80 字符截断 + `…`（matches `core/src/db.rs:SUMMARY_TRUNCATE_LEN = 80`）；GUI 端 `gui/src/stores/sessions.ts` 同样 80 字符。如果两端值不同，optimistic in-memory 跟 Rust 持久化值会 diverge 一字符，sidebar row 在 DB 写完成后会 jitter。

**`bump_session_after_turn` step_number 暂不持久化**：sessions 表没 `last_step_index` 列（is transient runtime field）。trait method 接收 step_number 但内部 `let _ = step` 抑制 unused warning；保留参数 future-proof for B4+ 加 audit table。

**Tests**: 47 new (76 → 123 total) — happy path / error envelopes / FK behavior / CASCADE on delete / summary truncation / bulk-variant partial-state handling。**关键**：`fresh_pool()` 必须 `PRAGMA foreign_keys = ON;`（SQLite 默认不开 FK），否则 FK violation test + delete_project SET NULL cascade test 都假阳。

**`list_projects` test 初版踩坑**：seed_project 默认 last_activity_at 都相同 `2026-05-19T00:00:00Z`，3 个 project 全同时间戳，ORDER BY 落到 implementation-defined → flaky test。修法 distinct timestamp。

### M4a addendum — set_session_llm method（commit `35f0e78`）

**为什么需要**：M4b 实施时发现 `runtimeStore.replaceLLMs` 通过 `mirrorSelectedLLMOnSession` helper 调用 `persistSession(snap)`（全行 UPSERT）持久化 LLM 选择。M4b 删 `persistSession` → dangling caller，没有 Rust endpoint 写 `sessions.llm_index` + `sessions.llm_display_name`。

**解决方案**：加 17th trait method `set_session_llm(id, index, display_name) -> SessionBrief`。pure UPDATE 走 sqlx，没有 `origin` 参数（replaceLLMs 在每次 bridge ready 触发 + 用户每次切 LLM 触发，audit signal 太 noisy）。

**为什么独立 commit**：playbook G5 警示「Rust trait method 加法独立 commit 不混 frontend 改动」。M4a 已 ship，临时把 addendum 跟 M4b 混反而违反隔离原则。独立小 commit (107 lines) + 3 test 干净。

## Rejected alternatives

- **M4a + M4b 单 commit 一气推**：1450 LOC Rust + 700 LOC frontend = 2150 LOC 单 commit 不可 review，违反 G5 trait method 隔离原则；中间 dogfood 颗粒度也被吞掉。强 reject。
- **`activateSession` 迁 sessionsStore**：编排 setActive + restoreTurns + spawnBridge，touches 三个 slice + `_runtimes`（M5 才迁）。强行迁到 sessionsStore 会让 sessions slice 依赖 messagesStore 概念。stays in useAppStore until M5 → 然后整段迁 sessionsStore 或 Rust event 驱动。
- **`patch_session` 通用 endpoint vs `set_session_llm` 显式**：通用 patch 接口跟其他显式 method (archive / rename / pin) 不一致；R6 risk register 写过 explicit endpoint 比 generic 好。
- **M4a addendum 跟 M4b 同 commit**：违反 G5 隔离。独立 commit 107 行轻量。
- **M4a 不写 cargo test，只写编译能过的 trait/impl**：47 个 test 既验证 happy path，又验证 NotFound / InvalidArgs / FK violation / CASCADE / 边界 truncation 逻辑——M4b 实施时如果 frontend code 把某个 invalid arg 形态传过来，test 已经定义了预期行为。
- **`bump_session_after_turn` persist `step_number` 进新列**：sessions 表 schema 加 last_step_index 列需 migration 008，nontrivial。改用 `let _ = step` 抑制 unused warning，future-proof signature。
- **保留 lib/db.ts 的 deleted functions 加 @deprecated** vs **B3-I6 直接 delete**：B3-I6 明文 "no @deprecated"，理由 B3 是 last hop，留就再也不删。强制执行。
- **sessions.ts 拆 subfolder pattern 在 M4b**：超 B3-I5 600 行硬上限（1054 行），但 mock fixtures + 17 action 全是顺序可读结构。M5 messagesStore 估计 600+ 行同样超限，到时一并切 `sessions/`, `messages/` 子目录 pattern (G11 警示)。M4b 不打断 momentum 优先 ship。

## Lessons learned

1. **M4 sub-plan 写作时仍然漏了 set_session_llm**：sub-plan R6 mention 了 project update partial 形态，但没 audit sessions LLM column 的现有持久化路径（runtimeStore.mirrorSelectedLLMOnSession → persistSession 全行 upsert）。M4b 实施时 grep `persistSession` 才暴露 dangling caller。**教训**：sub-plan audit 列「写过哪些字段 by 哪个路径」时，要 grep 现有持久化函数（persistSession / persistProject / loadSessions）所有 caller，确保 trait method surface 覆盖完整。M5 sub-plan 写作时对 messages 系列 (persistUserMessage / persistTurnEndToMessages / indexMessageFts / persistToolEventPending / persistToolEventApprovalDecision 等) 必须做同样 audit。
2. **Single-session push 全 M4 三阶段的真实代价 = 多 1 个 commit**：JC explicit「continue in same session」让 4 commit 一气 ship，质量保持的关键是 (a) sub-plan 锁定拆分策略减少 in-flight 决策 (b) M4a / addendum / M4b 三 commit 隔离 review boundary (c) 中间 cargo test 验证抓住 issue 不让其混进 M4b。但 N5 警示「疲劳期质量风险」仍成立——M4b 实施时漏 set_session_llm 是因为 sub-plan audit miss，不是 fatigue。
3. **`list_projects` test 时间戳碰撞踩坑** = 单元测试要主动 inject distinct fixture data，不能依赖 SQLite ORDER BY implementation-defined 行为。M5 写 messages 测试时（turn_index / sequence pair）同样注意。
4. **`PRAGMA foreign_keys = ON` 是 SQLite test setup must-have**：SQLite 默认 OFF，没开 FK violation test 全假阳。在 fresh_pool() helper 加这一行。
5. **`map_constraint_err` 字符串匹配是当前 sqlx 局限**：升级 sqlx 0.9 后 SqliteError typed error 改善（PR landed 2025），届时简化此 shim。
