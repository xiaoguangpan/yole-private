# YOLO Mode：审批是出口而非围栏

> Date: 2026-05-09（本日第二个 entry）
> Status: shipped — bridge / IPC / store / Settings / TopBar 全栈落地，55 个 bridge test 全过
> Related: [PRD §6.1 #4](../PRD.md) · [PRD §11.5](../PRD.md) · [DESIGN.md §4.1 TopBar](../DESIGN.md) · [DESIGN.md §9 Settings Approval](../DESIGN.md) · [docs/ipc-protocol.md §5.7](../ipc-protocol.md) · 上一篇 [2026-05-09 project model](./2026-05-09-project-model-coding-agent.md)

## Context

JC 在调研用户场景时意识到一个产品定位级的问题：**真实的 GA 用户里有相当部分把它当 coding agent 用**——这跟 Claude Code / Cursor / Aider 的用户重叠度很高。这群人对 approval fatigue 的容忍度极低（10-turn 会话点 N 个 file_patch 审批，没人受得了），如果 GA Workbench 不给"信任并跳过"的出口，他们会直接弃用 Workbench 回到裸跑 GA。

但 V0.1 PRD §6.1 七件事 #4 就是"用户可以审批高风险工具调用"，是 Workbench 的核心承诺之一。简单加一个 bypass toggle 等于把承诺削半。

讨论目的：决定 V0.1 加不加，加的话怎么做才**不破坏审批系统作为承诺**。

## Decisions

### 加，叫「YOLO mode」，进 V0.1

- **进 V0.1**：不进 V0.1 等于把 coding agent 用户推走；他们已经被 Claude Code 训练出"YOLO 出口"的肌肉记忆
- **命名 "YOLO mode"**：用户社区已接受这个 meme（Claude Code / Cursor / Aider 都用），自带自嘲意味——读到 "YOLO" 用户潜意识知道自己在做"反正都这样"的事；读到 "自动审批" 反而像聪明的功能，更易误用
- **PRD §6.1 #4 重新表述**：从"用户必须审批"改为"用户**可以**审批——审批系统是**出口**而非**强制围栏**"。这句重写本身就是产品定位的明确化，跟 Claude Code 的态度一致

### 颗粒度三层，V0.1 做粗 + 细两层

| 层级 | 入口 | 存活期 | V0.1 |
|---|---|---|---|
| Per call | Approval Card "Allow / Deny" | 单次 | ✅ 已有 |
| Always allow this tool | Approval Card "Always allow" | 永久 per tool | ✅ 已有 |
| **YOLO this session** | Approval Dock 一个按钮 | 当前 session 结束失效 | ❌ 推后 V0.2 |
| **YOLO mode（全局）** | Settings → Approval tab toggle | sticky，跨启动保留 | ✅ 本次新加 |

per-session 中间层故意 V0.1 不做——三个开关同时存在用户搞不清自己在哪层上，先把粗（全局）+ 细（per-call / per-tool）两层做扎实。

### TopBar persistent indicator（关键，没的话不能上线 V0.1）

YOLO 开启时 TopBar 必须有持续可见的 badge：

- 视觉：深琥珀（warning #C68762）背景 10% + 1px 同色边框 + ⚡ Phosphor `Lightning` thin + "YOLO" 12px Inter medium uppercase
- **不闪烁、不脉动**——动效会让用户疲劳后忽略，颜色对比足以传达"这是个状态，请知晓"
- 点击弹 Radix Popover：状态文案 + "立即关闭"深琥珀按钮 + "在 Settings 中查看 →" 链接
- 关闭后 indicator 立即消失

理由：人对自己的状态会忘。Claude Code 的 CLI 至少每次启动有 banner 警告——桌面 GUI 没有"启动时刻"的强警告，必须靠 persistent indicator 兜底。否则必出"忘了开着 YOLO，agent 跑了 git push --force"类事故。这条是 V0.1 release gate。

### Activation modal 文案精心设计

OFF → ON 弹 Radix Dialog：

- 标题：⚡ + "打开 YOLO 模式？"
- 主体：YOLO 全称解释 + bullet 列影响的工具 + 适用 / 不适用场景
- 二次确认按钮文案：「**是的，我知道在做什么**」——故意不写"确定"，避免条件反射点
- 取消按钮 default focus，按 Enter 默认是取消（再加一层防误触）

ON → OFF 不需要二次确认（关闭无害）。

### bridge 端走"硬跳过"而非"假审批"

两个候选：

- **方案 A**：desktop 收到 `tool_call_pending` 时如果 yolo on，立即发 `approval_response: allow_once`。bridge 不知道 yolo 概念
- **方案 B**：bridge 知道 yolo flag，`needs_approval` 第一行检查并直接 return False，不 emit `tool_call_pending`

选 **B**。理由：

- A 是 hack：每个 tool call 都来回一趟 IPC，bridge 阻塞等回复，network 延迟
- A 会污染 SQLite tool_events 表（每个都有 pending → allow 记录），Inspector Approvals tab 被刷屏
- B 干净：bridge 直接放行，没有 spurious pending events，tool_events 表只记真实的审批决策

### bridge spawn 后的 yolo state 同步

bridge 默认 `yolo_mode = false`。用户的持久化偏好通过 `set_yolo_mode { enabled: true }` IPC 命令同步：

- desktop 在 `ready` 事件 handler 里检查 store 的 yoloMode，为真则立即 send `set_yolo_mode`
- bridge 命令队列保证 spawn 后第一个 user message 之前 yolo state 已生效（顺序处理）
- 不需要 spawn args 加 `--yolo` flag——简化设计，统一走 IPC 通道

### 实现层关键决策

- **`yolo_check: Callable[[], bool]` 闭包**而非 `yolo: bool` 直接传入：与 always_allow_global / always_allow_project 的"传可变引用"模式一致；handler 在 dispatch 时调用 `self._yolo_check()` 读最新值。bridge 改 `state.yolo_mode = ...` 立即生效，不需要 rebuild handler
- **prefs API 加到 db.ts**（`getPref<T>` / `setPref<T>`）：JSON 序列化的 generic key/value 接口，覆盖未来所有 prefs 类需求（不只是 yolo_mode）。SQLite 不可用时 best-effort silent fail（Vite-only dev）
- **YOLO 优先级 > always_allow**：`needs_approval` 先检查 yolo（短路）再依次检查 approval_tools / always_allow。开 YOLO 时 always_allow 列表 dimmed 显示但不清空，关 YOLO 后立即恢复

### Settings UI：YOLO 块独立成 card

- 用 `border` + `rounded-[10px]` + `bg-surface` 框成独立 card，跟下方 per-tool 列表视觉分离
- YOLO 开启时 card 边框变 warning 色 + 浅琥珀背景 + 底部加一行"⚡ YOLO 已启用 · TopBar 显示状态"+ "立即关闭"按钮
- 下方 per-tool / always_allow 区域 YOLO 开启时整体 `opacity-50 pointer-events-none`，加一行 muted 提示"YOLO 已开启，下列规则当前不生效"

### 测试

5 个新 bridge test 覆盖核心路径：

1. YOLO on 时 approval tool 直接放行，不调 request_approval
2. 非 approval tool 在 YOLO 下行为不变
3. yolo_check 每次 dispatch 求值（不是 snapshot）—— flag 中途切换立即生效
4. YOLO 优先级覆盖 always_allow 列表
5. yolo_check 不传时默认 off（向后兼容）

加上 IPC `SetYoloModeCommand` 的两个 round-trip test（enabled true / false）。bridge 总计 55 个测试全过。

## Rejected alternatives

- **不加（保留"审批是强制围栏"定位）**：被否。用户行为已经证明这条规则跟真实使用场景冲突；强制围栏只会把人推向裸跑 GA，反而让 Workbench 失去价值
- **命名 "Auto-approve all" / "完全信任模式"**：太正经，用户开了不当回事；YOLO 自带降级语义更安全
- **YOLO 默认勾选 / 隐藏弹窗的 "记住选择"**：故意不做，每次首次开启都要看完文案
- **TopBar 用 toast 通知 YOLO 已开启**：toast 一闪即逝，3 秒后消失，无法兜底"忘了开着"。**必须 persistent indicator**
- **TopBar indicator 用动效闪烁 / 脉动**：用户疲劳后会自动忽略动态元素；静态颜色对比反而更可靠
- **TopBar indicator 用 error 红色**：开 YOLO 不是出错，是用户主动选择"我知道风险"。warning 深琥珀的语义更准
- **bridge spawn args 加 `--yolo` flag**：双通道（CLI flag + IPC）增加复杂度，IPC 单一通道已够
- **per-session YOLO 中间层 V0.1 一起做**：三层 toggle 同时存在用户搞不清自己在哪个开关上，分阶段做更稳
- **关闭 YOLO 同时清空 always_allow 列表**：两个 state 概念独立，user mental model 不应耦合。关 YOLO 应该回到"用户精心设置的 always_allow 状态"，而非"白板状态"
- **方案 A（desktop 端假审批）**：hack 设计，污染 tool_events 表，bridge 阻塞等 IPC 回复有延迟
- **PRD §11.4 / §11.5 顺序问题**：曾经先把 YOLO 写成 §11.4（在 Approval Card 之前），但 devlog 历史 ref 都指 §11.4 = Approval Card spec。改回 §11.4 = Approval Card / §11.5 = YOLO，保持历史 ref 稳定

## Open questions

- **YOLO 时 tool_call_pending 不 emit 的副作用**：Inspector Approvals tab 在 YOLO 下不会有新条目（这是设计意图），但用户可能 expect "看看 YOLO 期间发生了什么"。需观察用户内测反馈，必要时改成 bridge 仍 emit 但带 `auto_approved: true` 标记
- **YOLO 状态在 sidebar / Inspector 是否也要 indicator**：当前只在 TopBar。如果用户长时间隐藏 TopBar（macOS fullscreen 下 traffic light 消失，TopBar 仍可见但视觉边界模糊），是否需要 sidebar 顶部也有副 indicator？暂时不做，等真实抱怨
- **per-session YOLO 设计**：V0.2 候选时再细化。候选位置：Approval Dock 一个按钮 / Composer 旁一个 toggle 影响 next message 起的 turn
- **YOLO 状态进 IPC error 翻译**：如果 YOLO 期间某个 tool call 报错，error hint 是否要带"YOLO 已开启"上下文？暂时不需要，error message 跟 YOLO 状态正交
