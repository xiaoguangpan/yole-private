**Status**: Proposal — 设计完成，今 session 未实施
**Related**:

- 跟 YOLO mode（PRD §11.5）有深耦合
- 跟 [marathon session devlog](./2026-05-13-yole-rename-and-features-marathon.md) §14 同步

## Context

GA 官方 Streamlit 前端中文版 sidebar 有三个按钮：

```python
"开始空闲自主行动"  # 把 last_reply_time 设为 30 分钟前 → 立即触发
"⏸️ 禁止自主行动"
"▶️ 允许自主行动"
```

激活时 agent 自主跑：30 分钟无 user 活动 → 自动注入 `[AUTO]🤖 用户已经离开超过30分钟，作为自主智能体，请阅读自动化sop，执行自动任务。` agent 自己决定干啥（system prompt 里有"自动化 SOP"路径）。

**核心 tension**：Yole 当前定位是 human-in-the-loop 工作台。空闲自主行动把 human 拉出 loop——**不一定冲突**，可以定位为 "用户主动让出控制权的有限自主模式"——但需要 explicit 安全门。

## Decisions

### 1. Scope: 全局 toggle + 自动跟随 active session

**反对的**：per-session opt-in。讨论后判断"per-session"是过度防御——用户决策"我要离开"是一个**全局动作**，不是"给 session A 离开而不给 B 离开"。

**采用的**：全局 toggle，触发时只射 **当前 active session**。

```
[Sidebar Quick Actions]
[+ 新对话]
[🔍 搜索]
[🤖 自主行动]   ← 新增（V0.2）全局 toggle
```

行为：
- toggle 关：什么都不发生
- toggle 开：bridge 每 N 秒检查"当前 active session 最后一次 user message 时间"。超过阈值（30 min 默认）→ 给那个 active session 注入 auto-prompt
- 用户切 active session：idle timer 跟着切（切到 session B 后离开 30 min，就是 session B 触发）
- 关 toggle：立即停止 monitor

### 2. UI 位置 + 状态可见性

**主开关**：Sidebar Quick Actions 第三个按钮 "🤖 自主行动"（位置如上）。

**目标 session 提示**：sidebar 当前 active session row 上加一个小 🤖 indicator（仅当 toggle 开 + 此 session 是 idle target 时显示），让用户一眼看到"我离开后会在这里继续跑"。

### 3. 安全门：**必须开 YOLO mode 才允许自主**

最关键的安全决策。

讨论的三个选项：

- **a. 自主 = 必须 YOLO** ← 采用：透明 + 用户必须明确同意双重风险（YOLO + 自主）
- b. 自主 = 隐式 YOLO（自动 approve all）：隐藏 YOLO 行为，**否决**——状态不透明
- c. 自主 = 自动 deny 所有需审批的：agent 能跑但不能动危险工具，**否决**——很多有用工具会被锁死

**实施**：toggle 自主行动时检查 `yoloMode`，若关，提示 "需要先开 YOLO mode 才能开启自主行动" + 提供一键开启 YOLO 入口。

### 4. 预算硬限制

**单次触发硬限 N step（默认 30）**，到上限就停 + 系统消息「agent 自主跑了 N 步停下了，下面是它做的总结」。

完整 token / API 费 budget 推到 V0.3——做 step cap 已够大部分场景。

### 5. 阈值可调

- **默认 30 min**
- **范围 15 min ~ 4 hr**
- **位置**：Settings → Approval（跟 YOLO 同 tab）或新增 Settings → Behavior tab

global prefs `idle_autonomy_threshold_minutes`，hydrateFromDB 时读。

### 6. 通知与摘要

agent 半夜自动跑了 → 用户早上回来怎么知道？

- ✅ **a. 系统通知**（macOS notification 弹出 "Yole: agent 自主跑了 N 步" → 点击进 app）
- ✅ **b. Sidebar 该 session 用强提醒**：unread state + 额外 `🤖 自主跑过` label / icon
- ❌ c. 什么都不做：用户自己进来看 —— 不友好

**a + b 一起做**。

## IPC 协议增量

**新增 command**

```json
{
  "kind": "enable_idle_autonomy",
  "sessionId": "sess_xxx",   // 当前 active session
  "thresholdMinutes": 30
}
```

```json
{
  "kind": "disable_idle_autonomy",
  "sessionId": "sess_xxx"
}
```

**新增 event**

```json
{
  "kind": "idle_autonomy_fired",
  "sessionId": "sess_xxx",
  "elapsedMinutes": 32,
  "timestamp": "..."
}
```

agent 自主跑完后正常的 `turn_end` / `run_complete` 走标准路径——`idle_autonomy_fired` 只是个**通知用**事件，告诉 desktop "刚才那次 user_message 不是用户发的，是 idle 触发的"，desktop 借此做：

- 系统通知
- Sidebar `🤖 自主跑过` label
- 主 conversation 里用户消息气泡的视觉变体（apricot 边条 + 🤖 前缀，让用户清楚"这不是我说的"）

## 边界条件 + Open Questions

1. **YOLO 跟自主的双重开关**：用户先开 YOLO 后开自主 ✓；先开自主再开 YOLO ✓；开了自主后关 YOLO？应该**自动连带关掉自主**——单独 YOLO off 把 agent 推回审批模式，自主仍开就跑不下去。
2. **bridge 进程崩了**：idle monitor 是 bridge 端 thread，bridge 死了 monitor 也死，自主不再触发。用户回来看到 session 没动也没烦。可接受
3. **多 session 切换时的 timer 行为**：切到 session B 时是不是要立即重置 timer（"刚刚有活动"）？还是只重置 active session reference，仍按 last_user_message_time 计算？倾向后者——用户切换 session 本身不是"活动"，给 agent 写消息才是
4. **跟 Approval Dock 互动**：自主跑期间如果某个工具 still 走审批路径（比如 always_allow_global 没包含它），怎么办？YOLO 应该已经全跳过审批了。但若未来 YOLO 行为变化或加了 critical-tool override，需要处理。**当前假设**：YOLO on = 100% 跳审批，所以这个 case 不存在
5. **第一次开启时的引导**：第一次点 toggle，需要弹一个 modal 教育 "自主行动会让 agent 在你离开 30 分钟后自动继续。请确认你了解风险：[显示具体行为]" + dismiss 后不再弹。模仿 YOLO mode 第一次的 activation confirm modal

## 工程量预估 ~250 LOC

- Bridge 端 idle monitor thread（每 N 秒检查 last user message timestamp）：~50 LOC
- IPC: enable/disable command + idle_autonomy_fired event：~30 LOC
- Sidebar 🤖 indicator + Quick Action 按钮 + 关联 YOLO 检查：~50 LOC
- Settings 阈值 + 预算 cap UI：~30 LOC
- macOS 系统通知（Tauri 已有 API）：~25 LOC
- 主 conversation 自主消息视觉变体：~25 LOC
- 初次激活 confirm modal：~25 LOC

## 哲学定位

Yole 当前 PRD §6 V0.1 第 4 件事写：「审批是**出口**而非**强制围栏**」。

**空闲自主行动是这个哲学的延伸**——给"我信任 agent 自主跑"的用户一条**透明 + explicit + 可控**的路径：

- 永远不是默认开
- 永远全局（而不是悄悄 per-session 累积风险）
- 永远要走 YOLO 双重确认
- 永远有 step cap 兜底
- 永远有摘要反馈

不该做的："默默无声开启" / "per-session 各自有点小自主" / "AI 觉得 idle 就自己跑"——任何这种隐式自主都跟 Yole 整体定位冲突。

## Open Questions for 实施时

1. **idle monitor 在哪跑**？bridge 端最自然（已有 GA agent + history）。但 multi-session 下多个 bridge 各自跑 monitor 还是一个全局 monitor？倾向**每个 bridge 自己跑**——单 session 内自洽，且自主 toggle 实际状态可能在不同 bridge 之间漂（切 session 时）
2. **timer 用什么时钟**？wall clock vs monotonic？wall clock 受系统时间调整影响（用户改时区 / 自动同步跳变）。**monotonic** 更稳，但 idle "30 分钟"从 user 视角是 wall clock 概念。妥协：用 wall clock 算阈值，但保证 single-direction（时间倒退时不立即触发）
3. **agent 自主跑时 user 点 Stop 按钮 / 发 user message**？Stop 应能中断；新 user_message 应该当成新 conversation 继续。需要 idle monitor 重置 timer
4. **跟未来的 `/btw` `/branch` `/rewind` 冲突**？无——这几个是用户主动操作，跟 idle autonomy 是不同时刻
