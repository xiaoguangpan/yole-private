# 2026-05-15 — v0.1.0-alpha.2 Windows attach hotfix (fs:scope · mixed-separator joinPath · Settings GA Path editable)

**Date / Status**: 2026-05-15 / shipped (alpha.2 release)
**Related**: [v0.1 prod-build dogfood fixes](./2026-05-15-v0.1-prod-dogfood-fixes.md) (alpha.1 fixes that didn't cover Win-only failure modes) · [v0.1 ship and CI fallback](./2026-05-15-v0.1-ship-and-ci-fallback.md) (alpha.1 release SOP that this hotfix follows) · CLAUDE.md → Yole 架构原则 §1 (localhost only — informed the fs:scope analysis)

## Context

`Yole v0.1.0-alpha.1 · Windows (Alpha)` shipped to a small Win group on 2026-05-15. Within hours **two independent attach-time bugs** surfaced — both blocking Onboarding completion outright for affected users:

1. **User A**: GA installed under `~/Documents/GenericAgent` on Windows. Path picker selected the right folder; Onboarding's StepAttach still flagged "错误路径" (path validation failed). Manually appending a trailing `\` in the input box made it pass.

2. **User B**: GA installed at `D:\projects_2026\GenericAgent` (non-system drive). Screenshot showed the Health Check failing on the very first row ("GA 路径存在") with no path mutation helping.

Symptoms looked similar ("attach failed") so initial triage assumed one bug. They turned out to be two unrelated Windows-specific failures stacked on top of Mac-only assumptions in the codebase. macOS never surfaced either because:
- Mac dogfooders all happened to install GA at `~/Documents/GenericAgent` (lucky alignment with the scope allow-list)
- Mac path separators are `/` end-to-end, so `joinPath` worked by coincidence

Bundled as `v0.1.0-alpha.2` with a third shipped-together change: making the Settings → Runtime GA Path field manually editable (discussed independently earlier the same session, not bug-driven). Same release for atomic dogfood; release notes call out who needs to upgrade urgently vs. optionally.

## Decisions

### 1. Root causes per bug

| # | Reported symptom | Affected when | Root cause |
|---|---|---|---|
| A | Path picker returns a valid GA dir but validation fails; appending `\` fixes it | Windows, GA under `~/Documents` (passes `fs:scope`) | `lib/onboarding-validation.ts:joinPath` hardcoded `/` separator concatenation. Picker returns native `C:\Users\foo\GenericAgent`; `joinPath` produced `C:\Users\foo\GenericAgent/agentmain.py` (mixed separators). Tauri plugin-fs's `exists()` handles the mixed path inconsistently — the trailing `\` happened to nudge it into a path that worked. |
| B | "GA 路径存在" row red-X on a clearly real folder | Windows, GA outside the user-profile tree (`D:\`, `/opt`-style installs, external SSDs) | Tauri `fs:scope` in `capabilities/default.json` defaults to `$HOME/$DOCUMENT/$DESKTOP/$DOWNLOAD` only. Paths outside those globs throw on `exists()`; our `try/catch` reduces the error to `false`, reporting "not found" for a path that exists. |
| C (not a bug) | GA Path field in Settings → Runtime is picker-only | All platforms | Original design assumed picker is sufficient. JC asked to add manual typing earlier the same session — quality-of-life improvement, not user-reported breakage. |

### 2. Fixes shipped

**Bug A — Windows mixed-separator path join.** Routed both path expansion and join through `@tauri-apps/api/path`:

```ts
// before: macOS-only string concat
function joinPath(a: string, b: string): string {
  if (a.endsWith("/")) return a + b;
  return a + "/" + b;
}

// after: platform-native via Tauri path API
async function joinPath(a: string, b: string): Promise<string> {
  const { join } = await import("@tauri-apps/api/path");
  return await join(a, b);
}
```

Also added a `resolvePath` step that runs Tauri's `normalize` after `~`-expansion — collapses `..`, `.`, and crucially converts mixed separators to platform-native form. Five callers in `runHealthChecks` + one in `validateGAPath` add `await`. The previous `expandHome` rename to `resolvePath` reflects its expanded responsibility.

**Bug B — fs:scope blocking non-profile paths.** Added a one-line Rust Tauri command that bypasses plugin-fs entirely:

```rust
#[tauri::command]
fn path_exists(path: String) -> bool {
    std::path::Path::new(&path).exists()
}
```

JS-side `fsExists` switched from `@tauri-apps/plugin-fs::exists` to `invoke("path_exists", { path })`. Plugin-fs's `fs:scope` allow-list still applies to `writeTextFile` (used in `MessageActions.tsx` to save replies) but the picker-gated nature of that flow makes its scope failures user-visible-but-recoverable; the Onboarding scope blocker was a hard wall.

**Feature C — Editable GA Path in Settings → Runtime.** `PathField` now accepts an optional `onCommit` prop. When provided:
- Input is editable; debounced 300ms `validateGAPath` shows inline status (ok / missing-agentmain / not-found / checking)
- Commit triggers on Enter / blur; Esc reverts to saved value
- Picker button stays available; `onMouseDown(preventDefault)` prevents the blur-commit from racing with picker selection
- Only `not-found` blocks commit — `missing-agentmain` warns but commits (mirrors picker's lack of validation; trust the user for GA forks / non-standard layouts)

Python field stays readonly (picker disabled too) — V0.1 relies on `python-probe.ts`, which has its own constraints from the shell capability allow-list.

### 3. Why a custom Rust command rather than widening fs:scope to `**`

Three options on the table for Bug B:

**Option A — fs:scope = `**`**. One-line capability change. Concerns:
- Tauri 2's glob matching on Windows across drive letters (`C:\`, `D:\`, etc.) doesn't have a documented `**`-from-root semantics; behavior unknown without testing on each drive
- Opens **all** plugin-fs read/write surface in the future, not just `exists`. Hard to take back

**Option B — Custom Rust command**. ~10 lines of Rust, narrow surface (boolean existence only). Cross-platform separators handled by `std::path` natively. Aligns with [Yole 架构原则 §4 (path B Rust core authority)](../../CLAUDE.md) — this kind of capability is going to live in Rust core long-term anyway, doing it now is small-scope rehearsal.

**Option C — Drop the scope check by removing `fs:scope` from capabilities**. Equivalent to A but explicit. Same trade-offs.

Picked **B** because:
1. Yole is a trusted desktop tool with statically-bundled dist, no remote content load → the web-sandbox threat model `fs:scope` was designed for doesn't apply
2. The exposure is one narrow boolean read, not a full fs:read/write surface
3. JS callers were already routed through a single `fsExists` helper — surgical swap

### 4. Why ship all three changes in one release

Initial thought was to keep this purely a Win hotfix (Bug A + Bug B only) and ship the Settings editable input separately. Decided against:
- Settings editable input was already typecheck/lint-clean in the working tree; deferring meant a stale unmerged branch
- Same theme (GA Path handling); same touched files in part (validateGAPath reused by both Settings and Onboarding)
- Atomic dogfood: users picking up alpha.2 get one coherent set of changes to test rather than two separate alpha bumps

Trade-off accepted: alpha.2 is **not strictly a hotfix** — it includes a small feature. Release notes call this out by tier.

### 5. macOS-Intel from CI matrix

Dropped `macos-13` from `release.yml` matrix as a separate prep commit, motivated by:
- GitHub's macos-13 runner is on a deprecation track (no announced removal date, but flagged in their roadmap)
- JC's dev machine is an Intel Mac; local `pnpm tauri build --target x86_64-apple-darwin` covers any case where an Intel binary is needed (rare in alpha)
- No known alpha user is on Intel Mac

Method: `release.yml` matrix down to 2 entries (macos-14 + windows-latest); `release-workflow.md` updated to reflect new flow (Mac Intel = local build + optional `gh release upload` for the same Release page). Re-adding the matrix row is a 5-line revert if needed.

### 6. Release notes framing

Per [v0.1 ship and CI fallback devlog](./2026-05-15-v0.1-ship-and-ci-fallback.md) tiered-release pattern:

```
## Who needs to upgrade
- Windows alpha.1 users: critical — attach is completely broken in two common scenarios
- macOS alpha.1 users: optional — only relevant if GA is outside ~/Documents
```

Honest framing > "windows-patch" tag suffix, which would have mis-signaled Mac users with non-standard GA paths to skip the fix that also helps them.

## Rejected alternatives

- **Tag as `v0.1.0-alpha.1.windows-patch`** with Win-only artifact upload (JC's first instinct). Rejected because: (a) Mac users with GA outside `~/Documents` are silently affected by the same fs:scope bug; (b) `.windows-patch` suffix discourages them from upgrading; (c) CI matrix already builds Mac in parallel — "only Win" is more work, not less.
- **Widen `fs:scope` to `**`** as the Bug B fix. Rejected: opens future plugin-fs read/write surface beyond what we need; cross-platform glob behavior on Windows drive letters is undocumented.
- **Ship Win-only without dropping Mac Intel from CI**. Rejected: macos-13 is on a deprecation path anyway; doing the CI cleanup now while we're touching release config is cheaper than a separate prep commit later.
- **Keep `joinPath` as is and only fix `fsExists` routing**. Rejected: User A's report is unrelated to fs:scope; their GA was inside the allow-list. The mixed-separator bug stands on its own and would still bite users in similar configurations.
- **Defer Settings editable input to alpha.3**. Rejected: already-implemented and clean; atomicity benefit outweighs the "pure hotfix" framing.
- **Add a `validator` regex to `shell:allow-spawn` for Python paths** (parallel idea from alpha.1 prod-dogfood). Already established earlier that Tauri 2 stable doesn't support regex on `cmd`. Mentioned here because it's a path that doesn't exist for the path_exists fix either — `path_exists` is the analogous "narrow Rust command" move; precedent set.

## Open questions

- **MessageActions writeTextFile still goes through plugin-fs + fs:scope**. If a user picks a save location outside the allow-list (e.g. `D:\notes\reply.md`), it will silently fail with a console warning. Low-impact in v0.1 (save-reply is rarely used), but the symmetric fix is another narrow Rust command. Deferred until someone reports it.
- **Python probe doesn't auto-detect conda / pyenv / uv / asdf** Python on Windows. Same pre-registered-aliases constraint as Mac. V0.2 Rust-side spawn will fix this universally; not in alpha.2 scope.
- **conda / virtualenv layouts on D:\ drives** could still trip the python-probe step even after these fixes, because the probe's candidate list is path-prefix-based. Need to see if any Windows user reports a `D:\anaconda3\python.exe`-style setup.
- **No automated test catches the fs:scope hole**. The `bridge/tests/` matrix is Python; frontend has no E2E. A future regression on `path_exists` only surfaces in user dogfood. Acceptable for alpha; pre-V1 should grow at least a smoke test of `invoke("path_exists", ...)` returning true for a fixture path.

## Post-release CI maintenance

After alpha.2 publish, took the alpha.2 CI logs' two future-warnings as a prompt to clean up the workflow files. One commit covered three things:

**1. GitHub Actions Node 20 → 24 bump.** GitHub announced forced Node 24 by 2026-06-02 with full Node 20 removal 2026-09-16. Action maintainers had already shipped Node 24-ready majors — we were 2-4 majors behind on most. Bumps:

| Action | From | To | Notes |
|---|---|---|---|
| `actions/checkout` | v4 | v6 | Node 24 runtime |
| `actions/setup-node` | v4 | v6 | Node 24 runtime |
| `actions/upload-artifact` | v4 | v7 | v5 = Node 24, v7 adds optional direct-upload; our call shape unchanged |
| `actions/download-artifact` | v4 | v8 | v5 breaking only affects `artifact-ids` callers (we use names); v8 hash digest is cross-workflow concern (n/a in single-workflow upload+download) |
| `pnpm/action-setup` | v4 | v6 | v6 adds pnpm v11 support, still installs v10 when asked |
| `softprops/action-gh-release` | v2 | v3 | Pure Node 24 bump per maintainer note |

`swatinem/rust-cache@v2` and `dtolnay/rust-toolchain@stable` unchanged (same-major floating refs).

**2. `macos-13` dropped from `check.yml`.** release.yml dropped it at alpha.2; check.yml was inconsistent. Now both matrices mirror: `macos-15` aarch64 + `windows-latest` x64.

**3. macOS runner bump attempt: `macos-14` → `macos-26` (Tahoe), reverted to `macos-15` (Sequoia).** GitHub announced `macos-14` deprecation starting 2026-07-06 with full unsupport 2026-11-02. First push tried macos-26 (Apple's new year-based naming, GA on Actions) for longest runway. CI failed on `cargo check`:

```
error: unexpected argument 'check' found
Usage: rustup-init[EXE] [OPTIONS]
```

Root cause: `macos-26` pre-installs Rust 1.95.0 + rustup. `dtolnay/rust-toolchain@stable`'s PATH setup conflicts with the preinstalled binaries — `cargo` ends up shadowed by a `rustup-init` shim. Windows job in the same run passed cleanly, confirming the issue is isolated to the macos-26 × dtolnay interaction, not the action bumps.

Reverted to `macos-15` (Sequoia, GA since Sep 2025, well-tested with dtolnay) in a separate commit. macos-15 still satisfies the macos-14 Nov 2 deadline. Revisit macos-26 when:
- dtolnay/rust-toolchain releases a version that handles the preinstalled-rustup case, OR
- Yole moves to system Rust on the runner (post-B refactor, when we own more of the build infrastructure anyway)

**4. `windows-latest` floating tag kept as-is.** GitHub auto-redirects to `windows-2025-vs2026` on 2026-06-15. Pinning would shift maintenance burden to us without benefit; floating tag tracks GitHub's default which is exactly what we want.

Validation: `check.yml` ran on the maintenance push, macos-15 + windows-latest both green. Node 20 deprecation warning gone from CI logs.
